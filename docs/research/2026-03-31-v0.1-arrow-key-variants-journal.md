# Arrow Key Escape Sequence Variants Journal v0.1
**Date:** 2026-03-31
**Session:** c9510550-3bf3-4f87-bde0-ca10db1cfbab
**Status:** Published
**Preceding:** none
**Changes from v0.0:** Bug discovery and fix — two arrow key escape variants

## The Bug

Arrow keys stopped working in the "Restart previous session" picker. User could not navigate the dead session list. The lobby (main menu) worked fine.

## Root Cause

Terminal arrow keys send one of two escape sequences depending on the terminal mode:

| Key | CSI mode (normal) | SS3 mode (application cursor) |
|-----|-------------------|-------------------------------|
| Up | `\x1b[A` | `\x1bOA` |
| Down | `\x1b[B` | `\x1bOB` |
| Left | `\x1b[C` | `\x1bOC` |
| Right | `\x1b[D` | `\x1bOD` |

**CSI mode** (`\x1b[`) is the default. Terminals switch to **SS3 mode** (`\x1bO`) when "application cursor keys" is enabled — which programs like vim, less, and tmux do via the `DECCKM` escape sequence (`\x1b[?1h`).

PuTTY and other SSH clients respect the terminal's cursor key mode. When the user is in the lobby (no tmux attached), the terminal is in normal mode → `\x1b[A`. But if the user was previously in a session running Claude (which uses tmux, which enables application cursor keys), the terminal might stay in SS3 mode after detaching → `\x1bOA`.

## What Was Broken

The restart picker, user picker, and group picker only checked for `\x1b[A`:

```typescript
if (str === '\x1b[A') {  // only catches CSI mode
```

The lobby used `interactive-menu.ts` which checked both:

```typescript
if (str === '\x1b[A' || str === '\x1bOA') {  // catches both
```

So the lobby worked, but the restart picker didn't — for users whose terminal was in SS3 mode.

## The Fix

Added `|| str === '\x1bOA'` (and `\x1bOB` for down) to all three pickers:
- Restart picker (handleRestartInput)
- User picker (flow.step === 'users')
- Group picker (flow.step === 'groups')

Commit: `f0ee5f1`

## Broader Issue

This was a symptom of the duplicated input handling pattern identified earlier — 14+ separate arrow key handlers across the codebase, each potentially missing one variant. The widget system refactor (planned) would centralize this into a single key parser that normalizes both variants.

## Rule

**Every arrow key check in the codebase must test both `\x1b[X` and `\x1bOX` variants.** This applies to Up (A), Down (B), Right (C), and Left (D). No exceptions.

Grep to audit: `grep "x1b\[A" src/ | grep -v "x1bOA"` — any results are bugs.
