## Direction Decision

User approved the unified architecture: widgets + flow engine + renderers in one spec.

### Key Clarification: svg-terminal Relationship

The initial journal overstated the problem. svg-terminal already converts ANSI → xterm buffer → cells → spans → SVG. The lobby and creation flows *could* work through that existing pipeline — they'd just be terminal text rendered as SVG.

The real value of the widget system is **semantic UI data**. Instead of svg-terminal seeing "text with a > character on line 3," it knows "this is a list picker, cursor is on item 3, item 5 is disabled." This enables:
- Native HTML rendering in the browser (not a terminal emulator)
- SVG panels with hover states, touch targets
- Mobile-friendly layouts for the same flows
- Accessibility (screen readers understand lists, not ANSI)

Without widgets: everything works, but the lobby and creation wizard are "terminal screenshots" everywhere.
With widgets: they're first-class UI that each transport renders natively.

### Architecture Confirmed

```
Server (Node.js):
  Widget Layer → state machines, no rendering
  Flow Engine → chains widgets, evaluates conditions, collects results

  ANSI Renderer → for SSH clients (server-side rendering)
  Span Renderer → for svg-terminal (server-side rendering)

  WebSocket → sends structured widget state as JSON

Browser (JS):
  Receives JSON widget state
  Renders as HTML natively
  Sends back user actions as JSON
```

Three transports, one flow engine, one set of widgets. Browser is the only client-side renderer.

### Open Questions Resolved

1. **Widgets + flow engine together or separate?** Together. They're two halves of the same architecture.
2. **svg-terminal needs this?** Doesn't *need* it — existing pipeline works. But benefits from semantic data for native rendering.
3. **Browser JS communicates with flow engine?** Yes. JSON over WebSocket. Server sends widget state, browser sends back actions.
