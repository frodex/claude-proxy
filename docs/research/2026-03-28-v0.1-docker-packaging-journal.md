## Problem Statement

claude-proxy needs to be installable by anyone with `docker-compose up`. No manual setup, no system admin knowledge required. The current install requires: Linux server, Node.js, tmux, SSH keys, system users, systemd — too many moving parts.

## Key Design Decisions

### 1. One-command install
```bash
git clone && docker-compose up
```
Zero config to get started. First-run wizard handles everything.

### 2. Vault bundled for secrets management
HashiCorp Vault runs as a companion container. Pre-configured in dev mode for quick start, replaceable with production Vault for real deployments. Vault stores:
- SSH keys for remote hosts
- Anthropic API keys per user
- User credentials/mappings

### 3. Web-first onboarding
New users can't SSH in without keys. The web UI (port 3101) becomes the primary entry point:
- First user becomes admin
- Enter Anthropic API key → stored in Vault
- Generate or upload SSH keys
- Then SSH access works too

### 4. API key handling — the hard question
Claude Code needs Anthropic auth. Three options discussed:
- A) Each user enters their own key during onboarding → Vault stores it
- B) Admin enters one org key → shared by all
- C) Claude's built-in auth flow (prompts interactively)

Option C is current behavior but awkward in Docker (no browser for OAuth).
Option A is cleanest for multi-user.
Option B is simplest for teams with an org account.
Decision: Support all three. Default to A, offer B as a config option, C as fallback.

### 5. Trust model for distribution
Users won't run unknown code as root with their API keys. Mitigations:
- Container runs as non-root `claude-proxy` user
- API keys never touch disk — Vault only, injected as env vars
- Each user sandboxed by Unix permissions inside container
- Open source — auditable
- Docker isolation — no host filesystem access

## Architecture

```
docker-compose.yml
├── vault (hashicorp/vault)
│   ├── Port 8200 (internal only)
│   ├── KV v2 secrets engine
│   └── Pre-configured token for claude-proxy
│
└── claude-proxy (custom Dockerfile)
    ├── Port 3100 (SSH)
    ├── Port 3101 (Web + API)
    ├── Node.js 22 + tmux + Claude Code
    ├── Vault client (HTTP, reads keys on demand)
    ├── First-run setup wizard
    └── /data volume for persistent metadata
```

## Vault Integration Points

From Pippi's codebase (frodex/Pippi):
- Go client at `internal/services/vault/vault.go` — simple HTTP wrapper
- KV v2 at mount path `secret`
- Config: `VAULT_ADDR` + `VAULT_TOKEN` env vars
- Path pattern: `secret/<app>/<category>/<name>`

For claude-proxy:
```
secret/claude-proxy/ssh/ct100        → { private_key: "..." }
secret/claude-proxy/ssh/bigserver    → { private_key: "..." }
secret/claude-proxy/users/greg       → { ANTHROPIC_API_KEY: "sk-ant-..." }
secret/claude-proxy/users/frodex     → { ANTHROPIC_API_KEY: "sk-ant-..." }
secret/claude-proxy/config/org-key   → { ANTHROPIC_API_KEY: "sk-ant-..." }  (shared)
```

## What's In The Container

- Node.js 22 (LTS)
- tmux
- Claude Code binary (`/usr/local/bin/claude`)
- claude-proxy code (compiled TypeScript)
- SSH host keys (generated on first run, persisted in volume)
- tmux.conf (Ctrl+Q prefix, status off)
- launch-claude.sh (handles per-user launch)
- setup-remote.sh (for configuring remote hosts)

## What's NOT In The Container

- User home directories (created dynamically)
- API keys (in Vault, never on disk)
- SSH private keys for remotes (in Vault)
- Session data (in /data volume, survives container restarts)
