## Context

claude-proxy and svg-terminal run on the same box. Today claude-proxy's HTTP/WS API listens on **TCP 127.0.0.1:3101**. svg-terminal connects to it via `http://127.0.0.1:3101`. The question: should claude-proxy expose a **Unix domain socket** instead of (or alongside) a TCP port?

---

## The architecture we're working toward

```
                    INTERNET / LAN
                         │
                    ┌────▼────┐
                    │  nginx  │   (optional reverse proxy — TLS, single public port)
                    └────┬────┘
                         │
               ┌─────────▼──────────┐
               │    svg-terminal    │   port 3200 (or behind nginx)
               │                    │
               │  • 3D dashboard    │
               │  • login screens   │
               │  • session mgmt    │
               │  • terminal cards  │
               └────────┬───────────┘
                        │
                Unix domain socket
              /run/claude-proxy/api.sock
                        │
               ┌────────▼───────────┐
               │   claude-proxy     │   NO public TCP port
               │                    │
               │  • SSH :3100       │   (SSH stays TCP — that's its transport)
               │  • session manager │
               │  • tmux sessions   │
               │  • auth / user DB  │
               └────────────────────┘
```

**Key point:** Browsers never talk to claude-proxy directly. svg-terminal is the only network-facing web service. claude-proxy's API is internal.

## Why Unix sockets (vs TCP loopback)

### What a Unix domain socket is

A socket **file** on the filesystem (e.g. `/run/claude-proxy/api.sock`). Processes connect to it like a network socket, but traffic never touches the network stack. It's how PostgreSQL, Docker, Redis, and nginx communicate with local backends.

### Comparison

| Concern | TCP 127.0.0.1:3101 | Unix socket |
|---------|---------------------|-------------|
| **Network exposure** | Loopback only — but any local process can connect | No network exposure at all |
| **Access control** | Firewall rules or app-level auth | **Filesystem permissions** (owner, group, mode) — same UGO model claude-proxy already uses for tmux sockets |
| **Port conflicts** | Possible — another service might want 3101 | Not possible — it's a file path |
| **Performance** | Good (loopback avoids NIC but still goes through TCP stack) | ~10-30% less overhead — no TCP headers, checksums, or congestion control |
| **Discoverability** | Port scanners find it; `ss -tlnp` shows it | `ls -la /run/claude-proxy/api.sock` — visible only to filesystem access |
| **Debugging** | `curl http://127.0.0.1:3101/...` just works | `curl --unix-socket /run/claude-proxy/api.sock http://localhost/...` — slightly more verbose |
| **Browser access** | Browsers can connect directly | Browsers **cannot** — requires a proxy or bridge |

### Why it fits this project

1. **You already use this pattern.** `SocketManager` creates tmux sockets at `/var/run/claude-proxy/sessions/*.sock` with UGO permissions. The API socket is the same concept for the HTTP/WS server.
2. **svg-terminal is the web UI.** Browsers connect to svg-terminal, not claude-proxy. So claude-proxy doesn't need a network port for web traffic.
3. **Security by default.** A misconfigured firewall or `host: 0.0.0.0` won't accidentally expose the API to the network.
4. **SSH stays TCP.** SSH clients need a real network port (`:3100`). That doesn't change — SSH is its own transport.

## What needs to change in code

### claude-proxy (server side)

**`src/api-server.ts`** — `server.listen()` call:

```
Today:   server.listen(port, host)        →  TCP 127.0.0.1:3101
Target:  server.listen(socketPath)        →  /run/claude-proxy/api.sock
```

Both modes can coexist (listen on socket AND optionally on TCP for dev). Node.js `http.createServer` supports both natively.

**Config (`claude-proxy.yaml`):**

```yaml
api:
  socket: /run/claude-proxy/api.sock     # primary — svg-terminal connects here
  socket_mode: 0660                       # owner + group can connect
  socket_group: cp-services              # svg-terminal's service user is in this group
  # port: 3101                           # optional — uncomment for dev/debug
  # host: 127.0.0.1
```

**Startup:** Ensure `/run/claude-proxy/` exists, unlink stale socket file, listen, `chown`/`chmod`.

### svg-terminal (client side)

**`server.mjs`** — replace TCP references:

```
Today:   fetch('http://127.0.0.1:3101/api/sessions')
Target:  http.request({ socketPath: '/run/claude-proxy/api.sock', path: '/api/sessions' })
```

For **WebSocket bridge** (`ws` library supports `socketPath`):

```javascript
const upstream = new WebSocket('ws+unix:///run/claude-proxy/api.sock:/api/session/' + id + '/stream');
```

(Exact syntax depends on the `ws` library version — may need `{ socket }` option instead.)

### Permissions

| File | Owner | Group | Mode | Effect |
|------|-------|-------|------|--------|
| `/run/claude-proxy/api.sock` | `root` | `cp-services` | `0660` | root and cp-services group members can connect |
| svg-terminal service user | — | member of `cp-services` | — | svg-terminal can reach the API |

Same pattern as tmux session sockets. No new concepts.

## OAuth: layered approach (from NOTES)

### Design principle

**Claude-proxy must have a complete, standalone OAuth path.** If you run claude-proxy without svg-terminal, a user hitting `:3101` (or the TCP dev port) gets login → callback → cookie → API access — no svg-terminal dependency.

**svg-terminal hooks into that same flow** rather than reimplementing it. The layering:

```
Layer 0:  claude-proxy owns all auth logic (OAuthManager, resolveUser, cookie signing)
Layer 1:  claude-proxy serves its own minimal HTML for login/callback (standalone mode)
Layer 2:  svg-terminal can REPLACE that HTML with its own themed pages (hook mode)
```

### How it works

**Standalone (claude-proxy only, TCP dev port or reverse proxy):**

```
1. Browser hits /api/auth/login?provider=google
2. claude-proxy redirects to Google (callback URL = its own /api/auth/callback)
3. Google redirects browser back to /api/auth/callback
4. claude-proxy exchanges code, resolves user, sets cookie
5. claude-proxy redirects to / (serves minimal built-in HTML or a configured redirect URL)
```

No svg-terminal involved. Works today once OAuth is wired (per the existing wiring spec).

**With svg-terminal (Unix socket, production):**

```
1. Browser hits svg-terminal's login page (themed, 3D dashboard style)
2. Login page links to svg-terminal's /auth/login?provider=google
3. svg-terminal proxies to claude-proxy via Unix socket:
     POST /internal/auth/start { provider: "google", callbackUrl: "https://dashboard/auth/callback" }
4. claude-proxy returns the OAuth redirect URL (with svg-terminal's callback as redirect_uri)
5. svg-terminal redirects browser to Google
6. Google redirects browser back to svg-terminal's /auth/callback?code=...&state=...
7. svg-terminal forwards to claude-proxy via Unix socket:
     POST /internal/auth/exchange { provider: "google", code: "...", state: "..." }
8. claude-proxy exchanges code with Google, resolves user, returns identity + signed token
9. svg-terminal sets session cookie with that token
10. Browser is logged in, sees dashboard
```

**What claude-proxy adds for this:** Two **internal** endpoints (not exposed on TCP unless dev mode):

| Endpoint | Purpose |
|----------|---------|
| `POST /internal/auth/start` | Takes `{ provider, callbackUrl }`, returns `{ authUrl, state }` — the OAuth redirect URL with the caller's callback |
| `POST /internal/auth/exchange` | Takes `{ provider, code, state }`, does token exchange + `resolveUser`, returns `{ identity, token }` — signed payload svg-terminal puts in the cookie |

These reuse the existing `OAuthManager`, `GitHubAdapter`, and `resolveUser` — no new auth dependencies. The internal routes only listen on the Unix socket (or are gated by a header / source check on TCP dev mode).

### Themeable HTML (CSS hooks)

For **standalone mode** (no svg-terminal), claude-proxy serves its own login/callback/error pages. These should be **hookable**:

1. **Built-in HTML** uses semantic CSS classes (not inline styles):
   - `.cp-login-card`, `.cp-provider-btn`, `.cp-provider-google`, `.cp-provider-github`, `.cp-error-msg`, etc.
   - Default stylesheet: dark theme matching the SSH TUI palette (`#1a1a2e` / `#16213e`)

2. **Override mechanism** (optional, not required for v1):
   - Config: `api.auth.theme_dir: /path/to/custom/templates/`
   - If set, claude-proxy loads HTML from that directory instead of built-in
   - svg-terminal could provide a theme dir, but in practice won't — it renders its own pages and uses the `/internal/auth/*` API

3. **Why CSS classes matter even for internal pages:** If someone runs claude-proxy standalone with a reverse proxy, they might want to brand the login page. Semantic classes make that possible with a CSS override, not a template rewrite.

### What this avoids

- **No OAuth code in svg-terminal.** Zero new dependencies in svg-terminal for auth.
- **No shared npm module.** The `/internal/auth/*` API is the interface.
- **No duplicate token exchange.** claude-proxy does it once; svg-terminal gets the result.
- **No cookie format coupling.** claude-proxy signs the token; svg-terminal stores it opaquely. If the signing algorithm changes, only claude-proxy changes.
- **No nginx requirement.** svg-terminal handles the browser-facing routes; claude-proxy handles the provider-facing token exchange. No third service needed for routing.

## What stays TCP

| Service | Port | Why |
|---------|------|-----|
| **SSH** | 3100 | SSH protocol requires TCP — clients connect from the network |
| **svg-terminal** | 3200 (or 443 behind proxy) | Browsers need a network endpoint |
| **claude-proxy API** (dev only) | 3101 | Optional — `curl` convenience, standalone testing |

## Implementation effort

| Component | Size | Notes |
|-----------|------|-------|
| `api-server.ts` listen on socket | Small | ~10 lines: unlink stale, `listen(path)`, `chmod`/`chown` |

| Standalone login HTML (CSS-classed) | Small | ~50 lines of HTML with semantic classes + default dark stylesheet |
| svg-terminal `fetch` via socket | Medium | Replace `fetch()` calls with `http.request({ socketPath })` or a thin wrapper |
| svg-terminal WS bridge via socket | Medium | `ws` library socket path option |
| svg-terminal `/auth/login` + `/auth/callback` proxy routes | Medium | Thin handlers that call `/internal/auth/*` via socket |
| Socket permissions on startup | Small | Same pattern as `SocketManager` |

## Open questions

1. **Should TCP dev mode be a flag or always available?** Simplest: if `api.port` is set in YAML, listen on both socket and TCP. If not, socket only.
2. **Does svg-terminal need its own session cookie, or can it share claude-proxy's?** Recommendation: svg-terminal stores the **signed token from claude-proxy** as its cookie value. On each API call to claude-proxy, it passes the token in a header. One token format, two cookies (one per origin if needed).
3. **Does the `ws` library version in svg-terminal support Unix socket paths natively?** Needs a quick check of the installed version.
