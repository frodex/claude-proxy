# Partial Screen Data on Startup — Debugging Journal v0.2
**Date:** 2026-04-02
**Session:** Debugging svg-terminal partial screen on startup
**Status:** In Progress — root cause identified, fix options ready
**Preceding:** 2026-04-02-v0.1-partial-screen-on-startup-journal.md

---

## Symptom

After proxy restart, svg-terminal **full terminal views** (the interactive terminal windows, not thumbnails) show partial screen data. Terminals lower in the list (later to load) are more affected. Because terminal sessions receive sequential updates from that point forward, a partial initial load means the terminal is **permanently unusable** — subsequent delta updates build on the incomplete state. Only a forced redraw (resize) recovers the terminal.

**Distinct from thumbnail loading:** Thumbnails also load slowly under load, but that's expected — thumbnail updates are throttled and eventually catch up. The terminal issue is different: once a terminal gets partial initial data, it **never recovers** on its own because all subsequent data is relative to the incomplete screen state.

**Why this matters:** The terminal's xterm.js state machine processes ANSI sequences sequentially. If the initial state is partial (e.g., missing the first half of the screen), subsequent cursor movements, line edits, and scroll operations reference positions that were never established. The terminal is permanently desynchronized.

---

## Root Cause

**Race condition between tmux reattach and client connection.**

On proxy restart, `discoverSessions()` spawns 18 `tmux attach-session` processes in rapid succession (line 64 of index.ts). Each tmux attach needs time to:
1. Connect to the tmux server
2. Receive the current screen content
3. Pipe it through the PTY (node-pty)
4. Process through vterm (xterm.js headless)

This pipeline is **asynchronous** — process creation is sync, but data flow is async.

`transport.start()` and `startApiServer()` are called immediately after (lines 1008-1014). svg-terminal connects via WebSocket and requests screen state for all sessions (both thumbnails and full terminal views). The initial screen dump reads from `vterm.buffer.active` (pty-multiplexer.ts:367).

```
Timeline:
  t=0ms    discoverSessions() spawns 18 tmux attach processes
  t=1ms    transport.start() — SSH accepting connections
  t=2ms    startApiServer() — WebSocket accepting connections
  t=10ms   svg-terminal connects, opens terminal views for all sessions
  t=15ms   Sessions 1-3: tmux has sent full screen → vterm complete ✓
  t=15ms   Sessions 4-8: tmux partially sent screen → vterm partial ⚠
  t=15ms   Sessions 9-18: tmux barely connected → vterm empty/minimal ✗
```

**Why resize fixes it:** Resize triggers the existing code path that reads the tmux screen synchronously via `tmux capture-pane`, bypassing the async vterm buffer entirely.

**Why later terminals are worse:** Sessions are discovered and attached sequentially. Later sessions had less time for the async tmux→pty→vterm pipeline to complete before svg-terminal requests their state.

**Why it's permanent:** Unlike thumbnails (which periodically refresh and eventually catch up), full terminal views use a streaming delta model. The initial screen state is the baseline — all subsequent updates are relative cursor movements, line insertions, and character writes that assume the baseline is correct. A partial baseline means every subsequent update renders in the wrong position. The terminal state machine is permanently desynchronized until a full redraw resets the baseline.

---

## Evidence

1. 18 sessions active (6 default tmux + 12 socket-based)
2. `getBuffer()` reads from `vterm.buffer.active` — async-populated
3. No wait/ready mechanism between `discoverSessions()` and `transport.start()`
4. No "vterm settled" check before serving screen state to WebSocket clients
5. Resize works because it uses synchronous `tmux capture-pane -p`

---

## Fix Options

### Option A: Startup settle delay

Add a delay between `discoverSessions()` and `transport.start()`:

```typescript
sessionManager.discoverSessions();
await new Promise(resolve => setTimeout(resolve, 2000)); // wait for vterm to settle
transport.start();
```

**Pros:** Trivial, one line.
**Cons:** Arbitrary delay. Too short = still partial. Too long = slow startup. Doesn't help if tmux is slow (e.g., remote sessions). Doesn't help if clients connect after delay but vterm still isn't ready.

### Option B: Session ready flag with active settle detection

Track when each PtyMultiplexer's vterm stops receiving data (no onData for N ms = settled):

```typescript
// In PtyMultiplexer
private settled = false;
private settleTimer: NodeJS.Timeout | null = null;

// In onData handler:
this.settled = false;
if (this.settleTimer) clearTimeout(this.settleTimer);
this.settleTimer = setTimeout(() => { this.settled = true; }, 200);

isSettled(): boolean { return this.settled; }
```

WebSocket initial state send waits for settle:
```typescript
// Wait up to 3s for vterm to settle before sending initial state
const waitForSettle = async () => {
  for (let i = 0; i < 30; i++) {
    if (session.pty.isSettled()) return;
    await new Promise(r => setTimeout(r, 100));
  }
};
await waitForSettle();
const buffer = session.pty.getBuffer();
```

**Pros:** Adaptive — waits exactly as long as needed. Works for any number of sessions. Works for remote sessions too.
**Cons:** Adds 200ms minimum latency to first screen (settle timeout). More complex. Could delay initial render for sessions that ARE ready.

### Option C: Use tmux capture-pane for initial state (recommended)

Instead of reading from vterm buffer, use `tmux capture-pane -p -t <session>` for the initial WebSocket screen dump — the same mechanism that makes resize work:

```typescript
// Replace getBuffer() with synchronous tmux capture
const screenContent = session.pty.capturePane();
```

Add `capturePane()` to PtyMultiplexer:
```typescript
capturePane(): string {
  try {
    const cmd = this.socketPath
      ? `tmux -S ${this.socketPath} capture-pane -p -t ${this.tmuxId}`
      : `tmux capture-pane -p -t ${this.tmuxId}`;
    return execSync(cmd, { encoding: 'utf-8', timeout: 2000 });
  } catch {
    return ''; // fallback to empty if tmux not ready
  }
}
```

**Pros:** Synchronous — guaranteed complete. Uses the same mechanism that already works (resize). No timing guesses. No delays for sessions that are ready. Works for all session types (local, socket, remote via SSH).
**Cons:** `execSync` blocks the event loop briefly (~5-10ms per session). For 18 sessions connecting simultaneously, that's ~100-180ms of event loop blocking. Could be moved to async with `execFile` + Promise.

### Option D: Hybrid — vterm if settled, capture-pane fallback

```typescript
if (session.pty.isSettled()) {
  // Fast path — vterm has current state
  const buffer = session.pty.getBuffer();
  sendScreen(buffer);
} else {
  // Slow path — tmux hasn't finished sending data, read directly
  const content = session.pty.capturePane();
  sendScreen(content);
}
```

**Pros:** Best of both — fast when vterm is ready, correct when it isn't. No unnecessary `execSync` for sessions that are already settled.
**Cons:** More code paths. Settle detection still needed from Option B.

---

## Recommendation

**Option C (capture-pane for initial state)** for immediate fix — it's the simplest change that eliminates the race condition entirely. The `execSync` cost is acceptable for initial connection (not ongoing streaming).

**Then Option D as follow-up** — add settle detection and use capture-pane only as fallback, reducing `execSync` calls once sessions have had time to populate vterm.
