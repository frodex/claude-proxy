## Symptom

After proxy restart, svg-terminal **full terminal views** (the interactive terminal windows, not thumbnails) show partial screen data. Terminals lower in the list (later to load) are more affected. Because terminal sessions receive sequential updates from that point forward, a partial initial load means the terminal is **permanently unusable** — subsequent delta updates build on the incomplete state. Only a forced redraw (resize) recovers the terminal.

**Distinct from thumbnail loading:** Thumbnails also load slowly under load, but that's expected — thumbnail updates are throttled and eventually catch up. The terminal issue is different: once a terminal gets partial initial data, it **never recovers** on its own because all subsequent data is relative to the incomplete screen state.

**Why this matters:** The terminal's xterm.js state machine processes ANSI sequences sequentially. If the initial state is partial (e.g., missing the first half of the screen), subsequent cursor movements, line edits, and scroll operations reference positions that were never established. The terminal is permanently desynchronized.

---

## Root Cause

**Race condition between tmux reattach and client connection.**

On proxy restart, `discoverSessions()` spawns 18 `tmux attach-session` processes in rapid succession (line 64 of index.ts). Each tmux attach needs time to:
1. Connect to the tmux server
2. Receive the current screen content
3. Pipe it through the PTY (node-pty)
4. Process through vterm (xterm.js headless)

This pipeline is **asynchronous** — process creation is sync, but data flow is async.

`transport.start()` and `startApiServer()` are called immediately after (lines 1008-1014). svg-terminal connects via WebSocket and requests screen state for all sessions (both thumbnails and full terminal views). The initial screen dump reads from `vterm.buffer.active` (pty-multiplexer.ts:367).

```
Timeline:
  t=0ms    discoverSessions() spawns 18 tmux attach processes
  t=1ms    transport.start() — SSH accepting connections
  t=2ms    startApiServer() — WebSocket accepting connections
  t=10ms   svg-terminal connects, opens terminal views for all sessions
  t=15ms   Sessions 1-3: tmux has sent full screen → vterm complete ✓
  t=15ms   Sessions 4-8: tmux partially sent screen → vterm partial ⚠
  t=15ms   Sessions 9-18: tmux barely connected → vterm empty/minimal ✗
```

**Why resize fixes it:** Resize triggers the existing code path that reads the tmux screen synchronously via `tmux capture-pane`, bypassing the async vterm buffer entirely.

**Why later terminals are worse:** Sessions are discovered and attached sequentially. Later sessions had less time for the async tmux→pty→vterm pipeline to complete before svg-terminal requests their state.

**Why it's permanent:** Unlike thumbnails (which periodically refresh and eventually catch up), full terminal views use a streaming delta model. The initial screen state is the baseline — all subsequent updates are relative cursor movements, line insertions, and character writes that assume the baseline is correct. A partial baseline means every subsequent update renders in the wrong position. The terminal state machine is permanently desynchronized until a full redraw resets the baseline.

## Detailed: The Async vterm Pipeline

### What happens when `PtyMultiplexer.reattach()` is called

```
1. node-pty spawns: tmux attach-session -t cp-SESSION
   └── Creates a pseudoterminal (PTY) pair — master/slave fd pair
   └── tmux attach connects to the tmux server over this PTY

2. tmux server sees the new attach and sends the current screen content
   └── Raw ANSI: cursor positions, colors, text for all visible lines
   └── Flow: tmux server → PTY slave → PTY master → node-pty → onData callback

3. onData callback fires (possibly multiple times as data arrives in chunks):
   this.vterm.write(data)   ← xterm.js processes ANSI into internal screen buffer
   for (client of clients)  ← fans out to attached clients (none yet at startup)
```

**The reattach() method returns after step 1** (process spawned). It does NOT wait for steps 2-3 to complete. There is no callback, no promise, no event that says "tmux has finished sending the initial screen."

### What vterm.buffer.active contains at different moments

```
t=0ms   reattach() returns           → vterm buffer: empty (all blank lines)
t=5ms   first PTY data chunk arrives → vterm buffer: top 10 lines populated
t=10ms  second chunk arrives         → vterm buffer: top 30 lines populated
t=20ms  final chunk arrives          → vterm buffer: full 24x80 screen ✓
```

### What happens when svg-terminal requests the screen at t=10ms

```typescript
// api-server.ts line 743
const buffer = session.pty.getBuffer();  // reads vterm.buffer.active RIGHT NOW
```

At t=10ms, the client gets 30 lines. Lines 31-80 are blank. From this point on, all delta updates from the tmux session are **relative** — "cursor to line 45, write 'hello'" — but the client never received lines 31-80. Line 45 renders in the wrong place or on a blank line.

The terminal state machine is permanently desynchronized. Every subsequent ANSI sequence compounds the error.

### What "no vterm settled check" means

There is no mechanism to detect that vterm is still actively receiving initial data. The proxy serves whatever vterm has at the instant the client asks — it could be empty, partial, or complete, depending entirely on timing.

A settle check would detect the data flow and wait:

```
t=0ms   reattach() returns, settleTimer starts
t=5ms   onData fires → reset settleTimer (vterm still receiving)
t=10ms  onData fires → reset settleTimer (still receiving)
t=20ms  onData fires → reset settleTimer (still receiving)
t=220ms settleTimer expires (200ms with no new data) → mark as "settled"

t=300ms svg-terminal requests screen → isSettled() = true → serve full buffer ✓
```

Without the check, `getBuffer()` at t=10ms returns an incomplete screen and the proxy has no way to know it's incomplete.

### Why capture-pane bypasses this entirely

```typescript
execSync('tmux capture-pane -p -t cp-SESSION')
```

This asks the **tmux server directly** for the current screen content. The tmux server always has the complete screen — it's the authoritative source of truth. It doesn't matter whether the PTY→vterm async pipeline has finished. You're reading from tmux itself, not from vterm's lagging copy.

This is why resize already works — it uses capture-pane, which always returns the correct, complete screen regardless of vterm state.

## Fix Options

### Option A: Startup settle delay

Add a delay between `discoverSessions()` and `transport.start()`:

```typescript
sessionManager.discoverSessions();
await new Promise(resolve => setTimeout(resolve, 2000)); // wait for vterm to settle
transport.start();
```

**Pros:** Trivial, one line.
**Cons:** Arbitrary delay. Too short = still partial. Too long = slow startup. Doesn't help if tmux is slow (e.g., remote sessions). Doesn't help if clients connect after delay but vterm still isn't ready.

### Option B: Session ready flag with active settle detection

Track when each PtyMultiplexer's vterm stops receiving data (no onData for N ms = settled):

```typescript
// In PtyMultiplexer
private settled = false;
private settleTimer: NodeJS.Timeout | null = null;

// In onData handler:
this.settled = false;
if (this.settleTimer) clearTimeout(this.settleTimer);
this.settleTimer = setTimeout(() => { this.settled = true; }, 200);

isSettled(): boolean { return this.settled; }
```

WebSocket initial state send waits for settle:
```typescript
// Wait up to 3s for vterm to settle before sending initial state
const waitForSettle = async () => {
  for (let i = 0; i < 30; i++) {
    if (session.pty.isSettled()) return;
    await new Promise(r => setTimeout(r, 100));
  }
};
await waitForSettle();
const buffer = session.pty.getBuffer();
```

**Pros:** Adaptive — waits exactly as long as needed. Works for any number of sessions. Works for remote sessions too.
**Cons:** Adds 200ms minimum latency to first screen (settle timeout). More complex. Could delay initial render for sessions that ARE ready.

### Option C: Use tmux capture-pane for initial state (recommended)

Instead of reading from vterm buffer, use `tmux capture-pane -p -t <session>` for the initial WebSocket screen dump — the same mechanism that makes resize work:

```typescript
// Replace getBuffer() with synchronous tmux capture
const screenContent = session.pty.capturePane();
```

Add `capturePane()` to PtyMultiplexer:
```typescript
capturePane(): string {
  try {
    const cmd = this.socketPath
      ? `tmux -S ${this.socketPath} capture-pane -p -t ${this.tmuxId}`
      : `tmux capture-pane -p -t ${this.tmuxId}`;
    return execSync(cmd, { encoding: 'utf-8', timeout: 2000 });
  } catch {
    return ''; // fallback to empty if tmux not ready
  }
}
```

**Pros:** Synchronous — guaranteed complete. Uses the same mechanism that already works (resize). No timing guesses. No delays for sessions that are ready. Works for all session types (local, socket, remote via SSH).
**Cons:** `execSync` blocks the event loop briefly (~5-10ms per session). For 18 sessions connecting simultaneously, that's ~100-180ms of event loop blocking. Could be moved to async with `execFile` + Promise.

### Option D: Hybrid — vterm if settled, capture-pane fallback

```typescript
if (session.pty.isSettled()) {
  // Fast path — vterm has current state
  const buffer = session.pty.getBuffer();
  sendScreen(buffer);
} else {
  // Slow path — tmux hasn't finished sending data, read directly
  const content = session.pty.capturePane();
  sendScreen(content);
}
```

**Pros:** Best of both — fast when vterm is ready, correct when it isn't. No unnecessary `execSync` for sessions that are already settled.
**Cons:** More code paths. Settle detection still needed from Option B.

## Scale Analysis: 30 Browsers × 60 Sessions

Design target: 30 browser clients, each viewing 60 sessions (thumbnails + terminals). That's 1,800 WebSocket connections requesting initial screen state, potentially in a burst after proxy restart.

### The numbers

**Per initial screen dump:**
- vterm getBuffer() path: ~0.1ms (memory read, no I/O)
- capture-pane execSync path: ~5-10ms (fork process, exec tmux, read stdout, parse)
- capture-pane async path: ~5-10ms (same cost, doesn't block event loop)

**Startup burst — 60 sessions reattaching:**
- 60 tmux attach processes spawned
- vterm settle time: ~50-200ms per session (depends on screen complexity)
- All 60 settle within ~500ms (parallel, not sequential)

**Client connection burst — 30 browsers × 60 sessions:**
- Each browser opens 60 WebSocket connections
- 1,800 connections arrive over ~2-5 seconds (TCP + WS handshake)
- Each triggers an initial screen state request

### How each option fails or holds at scale

#### Option A: Startup settle delay — FAILS

A fixed 2-second delay works for 18 sessions. At 60 sessions, settle time is longer. At 60 sessions on remote hosts, settle time is unpredictable. The delay is a guess that gets worse as scale increases.

#### Option B: Settle detection with async wait — HOLDS but slow

Each of the 1,800 WebSocket connections polls `isSettled()` every 100ms. If vterm takes 500ms to settle, that's 5 poll iterations × 1,800 connections = 9,000 setTimeout callbacks in the first second. Node.js handles this fine, but every client sees 100-500ms latency on initial render.

At scale: **works but adds visible delay to every connection.**

#### Option C: capture-pane execSync — BREAKS

```
1,800 connections × execSync capture-pane = 1,800 sequential process forks
```

`execSync` blocks the event loop. Even at 5ms each, 1,800 calls = **9 seconds of blocked event loop**. During this time:
- No WebSocket messages processed
- No SSH data forwarded
- No terminal updates delivered
- All 30 browsers frozen

Even spread over 5 seconds of connection arrivals, that's still 360 execSync calls per second, each blocking for 5-10ms = **1.8-3.6 seconds of blocked time per second**. The event loop is effectively dead.

**Option C was my original recommendation. It does not scale.**

#### Option D: Hybrid vterm + capture-pane fallback — PARTIALLY HOLDS

After startup settle period (~500ms), most sessions have settled vterm buffers. The fast path (getBuffer, ~0.1ms) handles the majority. Only unsettled sessions trigger capture-pane.

But during startup burst (30 browsers connecting while 60 sessions are still settling), many sessions aren't settled yet. Worst case reverts to Option C behavior for those sessions.

At scale: **works after warmup, fragile during startup burst.**

### What actually works at 30×60

The core problem reframed: we need a way to serve the complete screen state for 60 sessions to 30 browsers without blocking the event loop and without serving stale data.

#### Option E: Async capture-pane with caching (new)

```typescript
// On reattach, schedule async capture-pane that populates a cache
private screenCache: string | null = null;
private cacheReady = false;

async warmCache(): Promise<void> {
  const cmd = this.socketPath
    ? ['tmux', '-S', this.socketPath, 'capture-pane', '-p', '-t', this.tmuxId]
    : ['tmux', 'capture-pane', '-p', '-t', this.tmuxId];
  
  const { stdout } = await execFileAsync(cmd[0], cmd.slice(1), { timeout: 2000 });
  this.screenCache = stdout;
  this.cacheReady = true;
}

// In onData handler — invalidate cache once vterm starts receiving real updates
this.screenCache = null;
this.cacheReady = false;
// (vterm is now live and getBuffer() is authoritative)

getInitialScreen(): { source: 'vterm' | 'cache'; data: any } {
  if (this.settled) {
    return { source: 'vterm', data: this.vterm.buffer.active };
  }
  if (this.cacheReady && this.screenCache) {
    return { source: 'cache', data: this.screenCache };
  }
  // Neither ready — return what vterm has (partial, but fast)
  return { source: 'vterm', data: this.vterm.buffer.active };
}
```

Flow at startup:
```
t=0ms     discoverSessions() spawns 60 tmux attach + 60 async warmCache()
t=5-50ms  warmCache() calls complete (async, parallel, non-blocking)
          → screenCache populated for all 60 sessions
t=100ms   browsers start connecting
          → getInitialScreen() returns cache (complete, no blocking)
t=200ms+  vterm starts receiving live data, cache invalidated
          → getInitialScreen() returns vterm buffer (now authoritative)
```

**At 30×60 scale:**
- 1,800 connections all served from in-memory cache — zero process forks
- getInitialScreen() is ~0.1ms per call (memory read)
- 1,800 × 0.1ms = 180ms total, no event loop blocking
- Cache warm happens once per session (60 async calls), not per client (1,800)

**After warmup:** vterm is live, cache is discarded, normal delta streaming takes over. The cache is only used during the startup window before vterm has settled.

#### Option F: Proactive screen snapshot on settle (new)

Instead of caching from capture-pane, snapshot vterm itself once it settles:

```typescript
// In settle handler
this.settleTimer = setTimeout(() => {
  this.settled = true;
  // Snapshot the now-complete vterm state as serialized screen
  this.settledSnapshot = this.serializeScreen();
}, 200);

getInitialScreen(): any {
  if (this.settled) return this.vterm.buffer.active;  // live
  if (this.settledSnapshot) return this.settledSnapshot;  // cached snapshot
  return this.vterm.buffer.active;  // partial, best effort
}
```

This avoids capture-pane entirely — no process forks ever. But it has a chicken-and-egg: the snapshot is only available AFTER settle, so clients connecting during settle still get partial data.

**Option F works well for steady-state** (clients connecting after startup), **but not for startup burst** (when clients and sessions race to be ready).

---

## Revised Recommendation

**Option E (async capture-pane with caching)** is the right long-term fix:

1. Zero event loop blocking — cache is populated asynchronously
2. Cache serves 1,800 clients from memory — one async fork per session, not per client
3. Self-invalidating — once vterm is live, cache is discarded
4. No arbitrary delays — clients get data immediately from cache or vterm, whichever is ready
5. Degrades gracefully — if cache isn't ready yet, falls back to partial vterm (same as today)

**Implementation cost:** ~30 lines in PtyMultiplexer + change getBuffer() callsite in api-server.ts to use getInitialScreen().

**Option B (settle detection)** should still be added as a foundation — Option E uses it to know when to stop serving cache and start serving vterm.

**Option C is withdrawn** — does not scale. Should not be used even as a fallback in the hot path.

## Full Lifecycle CPU Analysis

### How the current system works (no Option E)

To understand what Option E changes, first understand what happens today with zero changes.

#### The data path — one session, one active Claude

Claude is typing output in session A. Here's what the CPU does:

```
Claude writes to stdout
  → tmux server receives it, updates its screen buffer
  → tmux server sends it to all attached clients (our tmux attach process)
  → node-pty receives it via PTY fd, fires onData callback
  → SINGLE EVENT LOOP TICK handles everything:
      1. appendScrollback(buf)           ~0.01ms  (push to array)
      2. vterm.write(data)               ~0.1-1ms (xterm.js ANSI parsing)
      3. for each SSH client:
           client.write(buf)             ~0.01ms  (stream.write, buffered I/O)
      4. return — event loop free for next callback
```

Step 2 (vterm.write) is the expensive part — xterm.js parses ANSI escape sequences, updates cursor position, modifies the internal screen buffer. For a typical line of output this is ~0.1ms. For a large screen dump (clear + redraw) it can be ~1ms.

Step 3 is nearly free — `stream.write()` copies data to the kernel's TCP send buffer and returns immediately. It doesn't wait for the remote client to receive it.

**For WebSocket clients, the path is different.** They don't get raw PTY data. Instead:

```
onData fires → vterm.write(data) → onScreenChange callback marks dirty lines
  ... nothing happens yet ...
30ms later, setInterval fires for each WS client:
  → check if dirty (skip if not)
  → read dirty lines from vterm.buffer
  → convert each line to spans via lineToSpans()    ~0.05ms per line
  → JSON.stringify the delta                         ~0.05ms
  → ws.send(json)                                    ~0.01ms (buffered)
```

So WebSocket clients get batched updates every 30ms, with CPU cost proportional to the number of changed lines, not the raw byte count.

#### Scale: 60 sessions, all active simultaneously

In practice, not all 60 sessions output data at the same time. Claude sessions are idle most of the time (waiting for user input or thinking). When one session IS outputting, it typically produces short bursts (~100-500 bytes) with pauses between.

But worst case — 60 sessions all outputting simultaneously:

```
60 onData callbacks queue up in the event loop
Each takes ~0.1-1ms (vterm.write) + ~0.01ms × N_ssh_clients
Total: 60 × 1ms = ~60ms to process all callbacks

Meanwhile, 30ms setInterval fires for each WS client:
30 browsers × 60 sessions = 1,800 setInterval callbacks per 30ms window
Each dirty one: ~0.05ms × changed_lines + 0.05ms JSON + 0.01ms send
If 10 sessions are dirty: 1,800 × (10/60 dirty) × 0.2ms = ~60ms
```

That's ~120ms of CPU per 30ms window — the event loop can't keep up. In practice this doesn't happen because Claude output is bursty, not continuous.

**The single-thread answer: it matters at this scale, but not in the way you'd expect.** The event loop doesn't crash or drop data — it just falls behind. Callbacks queue up. The 30ms intervals fire late (every 50-80ms instead of 30ms). Clients see increased latency but not data loss. The system degrades gracefully because:
- SSH clients get data immediately (synchronous fanout in onData)
- WS clients get delayed deltas (their intervals fire late but still process all dirty lines)
- No data is dropped — it's all queued in the event loop

#### The actual bottleneck

It's not the onData fanout. It's the **1,800 setInterval callbacks** for WebSocket deltas. Each browser × each session = one timer. Most of these fire and immediately return (not dirty, skip). But even checking `if (!client.dirty) return;` 1,800 times per 30ms costs ~1-2ms of pure timer overhead.

Node.js timers are efficient (min-heap), but 1,800 concurrent timers is still significant. This is a pre-existing concern independent of Option E.

### What Option E adds and when it costs CPU

Option E only activates in two scenarios:

#### Scenario 1: Proxy restart (60 sessions reattach)

```
t=0ms    discoverSessions() runs synchronously
         For each of 60 sessions:
           spawn('tmux', ['attach-session', ...])   → OS creates process
           execFileAsync('tmux', ['capture-pane'])   → async, queued

         60 process spawns: ~60ms total (OS work, not event loop)
         60 async capture-pane calls: queued, will fire callbacks when done

t=10ms   Event loop is free. First capture-pane callbacks start arriving.
         Each: read stdout (~4KB screen) → store in screenCache
         Cost: ~0.1ms per callback × 60 = ~6ms total, spread over ~50ms

t=50ms   All 60 screenCache values populated.
         vterm.write() callbacks also starting to arrive from tmux attach.

t=100ms  First browser connects, requests 60 sessions.
         Each getInitialScreen(): read screenCache from memory → ~0.1ms
         60 sessions × 0.1ms = 6ms total. No process forks. No blocking.

t=200ms  vterm settles for most sessions. onData invalidates cache.
         Subsequent getInitialScreen() reads from vterm.buffer (also ~0.1ms).

t=300ms  Second browser connects. Same as above — vterm is live now.
         getInitialScreen() reads vterm.buffer. Zero Option E overhead.
```

**Total Option E CPU cost at startup: ~6ms** (60 cache reads when callbacks complete). After that, zero.

**Compare to current system at startup:** Browser connects, reads 60 incomplete vterm buffers. Same CPU cost (~6ms), but incorrect data. Option E costs the same but returns correct data.

#### Scenario 2: New session created while browsers are connected

```
User creates session "new-project"
  → createSession() → PtyMultiplexer constructor
  → spawn tmux new-session + spawn tmux attach-session
  → warmCache() fires async: one capture-pane call → ~5ms async, non-blocking

Browser 1 notices new session (lobby refresh), opens WS connection:
  → getInitialScreen() → cache is ready → serve from memory. Done.

Browser 2 connects 10 minutes later:
  → vterm has been live for 10 minutes, settled long ago
  → getInitialScreen() → serve from vterm.buffer. Cache is gone.
  → Zero Option E overhead.
```

**New session cost: one async process fork (~5ms, non-blocking).** Same whether 1 browser or 30 browsers connect to it.

#### Scenario 3: Steady state — terminals active, browsers connected

```
30 browsers connected. 60 sessions. 5 sessions have active Claude output.

Per output burst from one session:
  onData fires:
    appendScrollback          → same as today
    vterm.write               → same as today
    SSH client fanout         → same as today
  
  Option E adds: nothing.
  Cache was invalidated long ago. settled=true. 
  getInitialScreen() is never called (only on NEW connections).
  
  The 30ms WS delta intervals: same as today. Option E is invisible.
```

**Steady-state Option E overhead: exactly zero.** The cache and settle timer only matter at connection time. Once a WS client has its initial screen and is receiving deltas, Option E has no code in the hot path.

#### Scenario 4: Terminal added to a browser mid-session

```
User opens a terminal view for session X in their browser.
  → New WS connection for that session.
  → getInitialScreen() called once:
      vterm settled (session has been running for hours) → read vterm.buffer
      Cost: ~0.1ms. Same as today.
  → 30ms delta interval starts. Same as today.
```

**No difference from current system.** Option E's cache is only relevant for sessions that haven't settled yet.

#### Scenario 5: Terminal removed from a browser

```
User closes a terminal view.
  → WS connection closes.
  → clearInterval(pollTimer) — one fewer 30ms timer.
  → session.pty.onScreenChange callback unregistered.
  → No Option E involvement.
```

**No difference from current system.**

### Summary: Option E CPU cost vs current system

| Scenario | Current system | With Option E | Difference |
|----------|---------------|---------------|------------|
| **Proxy restart, 60 sessions** | 60 vterm reads (partial data) | 60 async forks + 60 cache reads | +6ms async (non-blocking), correct data |
| **New session created** | vterm read (possibly partial) | 1 async fork + cache | +5ms async (non-blocking), correct data |
| **Browser connects to existing session** | vterm read | vterm read (settled) | Zero |
| **Active Claude output, 30 browsers** | onData + 1800 delta intervals | identical | Zero |
| **Terminal opened mid-session** | vterm read | vterm read (settled) | Zero |
| **Terminal closed** | cleanup | cleanup | Zero |
| **Idle system** | 1800 timers checking dirty=false | identical | Zero |

**Option E adds CPU cost only when sessions are new or reattaching — one async fork per session, not per client. In steady state it's invisible. The single-thread event loop is never blocked.**

### Does single-threading matter?

For Option E specifically: **no.** The async capture-pane forks a child process (OS handles it on a separate thread/core), the callback fires when it's done (microseconds of event loop time to store the result). The event loop is never blocked.

For the overall system: the existing bottleneck is the 1,800 setInterval timers for WS delta delivery (30 browsers × 60 sessions). That's a pre-existing O(browsers × sessions) cost that runs every 30ms. Option E doesn't make this better or worse. If this becomes a problem, the fix is to coalesce timers (one timer per browser that batches all session deltas, or one timer per session that fans out to all browsers) — but that's a separate concern from the initial-screen-data bug.
