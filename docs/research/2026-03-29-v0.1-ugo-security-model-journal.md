## The Problem

claude-proxy currently enforces session access control at the application layer. The proxy checks its own ACLs (hidden, public, allowedUsers, allowedGroups) when a user tries to join a session. But the underlying tmux sessions run as root (or a configured `runAsUser`), and anyone with shell access can `tmux attach` directly, bypassing all proxy protections.

The proxy is the only security boundary. If the proxy has a bug, all sessions are exposed.

## Design: Unix Permissions As The Security Boundary

### Core Principle

OAuth identity maps to a real Linux user. Sessions run as that user. Unix permissions enforce isolation. The proxy is a UI layer, not the security layer.

### Session Ownership

Every tmux session runs as the user who created it. The proxy uses `su` to launch tmux as that user. The tmux server socket is owned by that user with permissions that control who can attach.

```
/var/run/claude-proxy/sessions/
  myproject.sock          owner=greg  group=cp-sess-myproject  mode=0770
  private-work.sock       owner=greg  group=greg               mode=0700
  team-debug.sock         owner=greg  group=cp-devteam         mode=0770
```

### Permission Levels Via Socket Mode

- **Private session:** socket mode `0700` — only the owner can attach
- **Team session:** socket group = team group (e.g. `cp-devteam`), mode `0770` — group members can attach
- **Public session:** socket group = `cp-users` (all registered users), mode `0770`

### tmux With Custom Sockets

tmux supports `-S /path/to/socket` for custom socket paths. The proxy launches tmux with:

```bash
su - greg -c "tmux -S /var/run/claude-proxy/sessions/myproject.sock new-session -d -s cp-myproject ..."
```

And attaches users with:

```bash
su - frodex -c "tmux -S /var/run/claude-proxy/sessions/myproject.sock attach-session -t cp-myproject"
```

If frodex isn't in the right group, the OS returns permission denied. The proxy doesn't need to check — the OS did it.

### Group Management

Groups control access. The proxy manages them with standard Linux commands:

| Operation | Command | When |
|-----------|---------|------|
| Create session group | `groupadd cp-sess-myproject` | Session creation (if shared) |
| Add user to session | `usermod -aG cp-sess-myproject frodex` | Owner grants access |
| Remove user from session | `gpasswd -d frodex cp-sess-myproject` | Owner revokes access |
| Delete session group | `groupdel cp-sess-myproject` | Session destroyed |

Team groups (e.g. `cp-devteam`) are managed separately — one group change affects all sessions that use that group.

### Live Permission Changes

When the owner changes permissions on a live session with connected users:

**Adding a user:** Works immediately. Proxy mediates all attaches via `su`, which picks up current group membership.

**Removing a user:** Two steps:
1. Remove from group (`gpasswd -d frodex cp-sess-myproject`) — prevents new attaches
2. Detach active connection (`tmux -S /path/socket detach-client -t /dev/pts/X`) — kicks current session

**Why both steps:** Unix doesn't revoke access to already-open file descriptors when permissions change. The group change prevents re-entry; the detach handles the live connection.

### Automatic Enforcement: inotify on /etc/group

Problem: if someone changes group membership outside the proxy (e.g. `gpasswd` on the command line), the proxy doesn't know.

Solution: `fs.watch('/etc/group')` — Linux kernel inotify feature. When `/etc/group` is modified, the kernel notifies the proxy immediately (sub-second). The proxy then:

1. Re-reads group memberships for all connected clients
2. For each active session, checks if connected users still have access
3. Detaches anyone who lost access via `tmux detach-client`

```typescript
import { watch } from 'fs';

watch('/etc/group', () => {
  // /etc/group changed — re-evaluate all active session memberships
  for (const session of activeSessions) {
    for (const client of session.clients) {
      if (!userCanAccessSocket(client.username, session.socketPath)) {
        tmuxDetachClient(session, client);
      }
    }
  }
});
```

No polling. No CPU cost while idle. Catches all group changes regardless of source.

### Proxy ACLs Become UI Hints

The current proxy ACL system (hidden, public, allowedUsers, etc.) doesn't go away — it becomes the UI layer:

- **hidden** — session doesn't appear in lobby/API for non-owners (but the socket still exists)
- **public/private** — controls which group the socket belongs to
- **allowedUsers/Groups** — proxy sets up the corresponding Unix groups
- **password** — proxy-level gate before attaching (defense in depth, not the primary boundary)
- **viewOnly** — proxy intercepts input for non-owners (still application-level, but input blocking not access blocking)

The settings UI writes to Unix groups. The lobby reads from Unix groups (via socket permissions). The proxy is a translator between the UI and the OS, not an enforcer.

### Auth Stack

```
SSH Client:
  SSH public key → Linux username → su to user → tmux socket permissions

Web Client:
  OAuth (Cloudflare Access) → JWT → Linux user mapping → su to user → tmux socket permissions

Both paths converge at: Linux user → Unix permissions
```

### User Provisioning (OAuth)

When a new OAuth user first connects:
1. Cloudflare Access validates identity, issues JWT
2. Proxy extracts identity from JWT
3. Checks for mapped Linux user
4. If none: creates system account (`useradd`), home directory, adds to `cp-users` group
5. Installs claude if needed (`curl -sL https://claude.ai/install.sh | bash` as the new user)
6. Maps OAuth identity → Linux username (persisted in proxy config)
7. All subsequent requests execute as that Linux user

### Migration Path

Current state → UGO model:

1. **Add socket-based tmux launch** — `-S` flag, socket directory, permissions
2. **Default `runAsUser` to authenticated user** — not root
3. **Map proxy ACL changes to group operations** — when owner edits allowedUsers, proxy runs `usermod`/`gpasswd`
4. **Add inotify watch on /etc/group** — reactive enforcement
5. **Add `cp-users` group** — all registered users, used for "public" sessions
6. **Proxy ACLs remain as UI layer** — lobby visibility, not enforcement

## OAuth Providers

Initial implementation: three providers.

| Provider | Type | Notes |
|----------|------|-------|
| Google | OIDC-compliant | Config-only, discovery URL, just works |
| Microsoft | OIDC-compliant | Config-only, enterprise/corporate users |
| GitHub | OAuth 2.0 (not OIDC) | Needs ~50 lines custom adapter, no discovery URL |

Future: local/self-hosted solution (Keycloak or similar) for orgs that don't want external providers.

OIDC providers are add-from-config — put discovery URL + client ID + secret in `claude-proxy.yaml`, they work. GitHub is a built-in special case.

## Open Questions

1. **Socket directory location:** `/var/run/claude-proxy/sessions/` (tmpfs, cleared on reboot) or `/srv/claude-proxy/sockets/` (persistent)? Sessions survive reboots via tmux, but sockets don't survive reboots anyway (tmux recreates them). Probably `/var/run/` is correct.

2. **One tmux server per user or shared?** Currently all sessions use the default tmux server. With `-S`, each session gets its own server (socket). This means `tmux list-sessions` doesn't show all sessions — the proxy must scan the socket directory instead. This changes `discoverSessions()`.

3. **viewOnly enforcement:** This stays application-level — the proxy intercepts input from non-owner clients. Unix permissions don't have a "read but not write" model for tmux sockets. Could use a named pipe or read-only shell wrapper, but that's complexity for little gain.

4. **Remote sessions:** Remote tmux sessions run on another host. The socket is on the remote host. Group management must happen on both the proxy host (for UI) and the remote host (for enforcement). This adds complexity — worth deferring?
