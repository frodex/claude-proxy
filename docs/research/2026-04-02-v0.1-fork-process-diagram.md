# Fork Session Process — Process Tree Diagram v0.1
**Date:** 2026-04-02
**Session:** Debugging svg-terminal nested tmux issue
**Status:** In Progress
**Preceding:** none
**Changes from v0.0:** Initial — documents the full process chain when forking a session

---

## The Problem

When forking from within a running session, svg-terminal may see "windows looking at windows" — nested tmux attach chains. This document traces every process spawned to identify where nesting occurs.

---

## Process Architecture: Normal Session

When a session is created, here's the process tree:

```
claude-proxy (node, PID 1)
  └── node-pty (spawns a PTY pair)
        └── tmux attach-session -t cp-SESSION_NAME   ← PtyMultiplexer.attachToTmux()
              └── (attaches to existing tmux server session)

tmux server (separate process, owns the actual terminal)
  └── tmux session: cp-SESSION_NAME
        └── window 0
              └── /bin/bash /tmp/claude-proxy-launch-cp-SESSION_NAME.sh  ← launch script
                    └── su - root -c 'cd /workdir && launch-claude.sh --resume'
                          └── claude --resume
```

**Two separate process trees:**
1. **tmux server** — owns the session, runs `claude` inside a bash→su chain
2. **claude-proxy** — attaches to that session via `tmux attach-session` through node-pty

The SSH client (or svg-terminal) connects to claude-proxy, which pipes bytes between the SSH connection and the node-pty (which is attached to tmux).

```
SSH client ←→ claude-proxy ←→ node-pty ←→ tmux attach ←→ tmux server ←→ bash ←→ su ←→ claude
```

## Process Architecture: Fork

When user presses Ctrl+B f in a session:

### Step 1: Detach from source session

```
claude-proxy
  ├── node-pty (PTY for source session) ← client DETACHED here
  │     └── tmux attach-session -t cp-SOURCE
  │           └── (source tmux session keeps running)
  │
  └── (client is now seeing the fork form rendered by claude-proxy directly)
```

The client's SSH bytes are now routed to `handleForkInput()` in claude-proxy, NOT to the tmux session. The source session keeps running — other clients can still see it.

### Step 2: User submits fork form → executeFork()

```python
# What executeFork() calls:
this.createSession(
  forkName,          # "SOURCE-fork_01"
  client,            # the SSH client
  session.runAsUser, # same user as source
  'claude',
  access,
  ['--resume', claudeId, '--fork-session'],  # ← key args
  undefined,         # no remote
  session.workingDir,
)
```

### Step 3: createSession() spawns new tmux session

**New launch script** written to `/tmp/claude-proxy-launch-cp-SOURCE-fork_01.sh`:
```bash
#!/bin/bash
rm -f "/tmp/claude-proxy-launch-cp-SOURCE-fork_01.sh"
su - root -c 'mkdir -p /workdir && cd /workdir && /srv/claude-proxy/scripts/launch-claude.sh --resume <CLAUDE_SESSION_ID> --fork-session'
```

**tmux command** (from PtyMultiplexer constructor):
```bash
tmux -S /var/run/claude-proxy/sessions/cp-SOURCE-fork_01.sock \
  new-session -d \
  -s cp-SOURCE-fork_01 \
  -x 80 -y 24 \
  /tmp/claude-proxy-launch-cp-SOURCE-fork_01.sh
```

This creates a **detached** tmux session (`-d` flag). Claude starts inside it.

### Step 4: PtyMultiplexer attaches to new session

Immediately after `new-session -d`, the constructor calls `attachToTmux()`:

```bash
# node-pty spawns:
tmux -S /var/run/claude-proxy/sessions/cp-SOURCE-fork_01.sock \
  attach-session -t cp-SOURCE-fork_01
```

### Step 5: joinSession() pipes client to new PTY

```
claude-proxy
  ├── node-pty (PTY for source session) ← still running, other clients may be here
  │     └── tmux attach -t cp-SOURCE
  │
  ├── node-pty (PTY for fork) ← CLIENT ATTACHED HERE
  │     └── tmux attach -t cp-SOURCE-fork_01
  │           └── (new tmux session running claude --resume <id> --fork-session)
  │
  └── client SSH bytes now route to fork PTY
```

### Step 6: leaveSession() detaches from source

```python
this.leaveSession(sessionId, client)  # remove from source session's client map
```

## Full Process Tree After Fork

```
tmux server (default)
  └── cp-SOURCE (session)
        └── bash → su → claude --resume              ← ORIGINAL, still running

tmux server (socket: cp-SOURCE-fork_01.sock)
  └── cp-SOURCE-fork_01 (session)
        └── bash → su → claude --resume <id> --fork-session  ← FORK, new process

claude-proxy (node)
  ├── node-pty → tmux attach -t cp-SOURCE              ← source PTY (may have 0 clients)
  ├── node-pty → tmux attach -t cp-SOURCE-fork_01      ← fork PTY
  │                                                      ↑ client SSH is connected here
  └── SSH transport ←→ client
```

## Where Nesting Could Happen

### The chain for svg-terminal rendering:

```
svg-terminal ←[SSH]→ claude-proxy ←[node-pty]→ tmux attach ←[tmux protocol]→ tmux server → claude
```

This is **3 layers of terminal emulation**:
1. **svg-terminal** renders ANSI from the SSH stream
2. **tmux attach** is a terminal client that reads from the tmux server
3. **tmux server** is a terminal multiplexer with its own screen buffer

When claude-proxy writes the fork form (Step 1 above), it writes ANSI directly to the SSH client — no tmux involved. But when the fork session starts (Step 5), the bytes flow through tmux again.

### Potential "windows looking at windows" scenarios:

**A. tmux attach inside tmux** — If somehow the `tmux attach-session` command runs *inside* another tmux session rather than directly from node-pty, you'd get nested tmux. This shouldn't happen because node-pty spawns `tmux attach` directly, not inside a shell that's inside tmux.

**B. Screen buffer stale content** — When detaching from source and attaching to fork, the xterm.js headless terminal (`vterm`) in PtyMultiplexer may still contain the source session's screen content. The fork session starts fresh but if svg-terminal captures the transition, it might briefly see the old content behind the new.

**C. tmux status bar stacking** — Both the source and fork tmux sessions have their own status bars. If tmux.conf has a status bar enabled, and the tmux attach also shows a status bar, you'd see two status bars. The proxy's `tmux.conf` should disable the status bar (`set -g status off`).

**D. The detach/attach gap** — Between `session.pty.detach(client)` in startForkFlow (Step 1) and `joinSession()` in createSession (Step 5), the client is in a limbo state where claude-proxy renders the form directly. If svg-terminal is trying to maintain a persistent terminal state, this mode switch (tmux output → direct ANSI → tmux output) could cause rendering artifacts.

## BUG FOUND: clientState not updated after fork

### Reproduction

1. Open session A in svg-terminal (via claude-proxy SSH)
2. SSH into claude-proxy from a separate shell
3. Join session A (now two clients: svg-terminal + SSH)
4. Fork session A via Ctrl+B f → complete form → confirm
5. SSH client is now "in session B" — sees session B output
6. Type in SSH — **actions mutate session A** (visible in svg-terminal)

### Root cause

`executeFork()` runs inside `session-manager.ts`. It calls `createSession()` which internally calls `joinSession()` — this attaches the SSH client to session B's **PtyMultiplexer** (output routing is correct). Then `leaveSession()` removes the client from session A's client map.

But `clientState` in `index.ts` is **never updated**:

```
AFTER FORK:
  clientState[ssh-client].sessionId = 'cp-A'   ← STALE, should be 'cp-B'
  Session B PTY has ssh-client attached          ← output goes to client (correct)
  Session A PTY has svg-terminal attached         ← output goes to svg (correct)
```

When the user types:

```typescript
// index.ts onData handler:
const state = clientState.get(client.id);  // → { sessionId: 'cp-A' }
sessionManager.handleInput(state.sessionId, client, data);  // routes to SESSION A!
```

**Input goes to session A, output comes from session B.** The user sees session B's screen but types into session A. svg-terminal (watching session A) shows the keystrokes.

### Why normal creation doesn't have this bug

The normal creation flow stays in `index.ts` the whole time:

```typescript
// index.ts finalizeSessionFromResults():
const session = sessionManager.createSession(...);
const state = clientState.get(client.id);
state.mode = 'session';
state.sessionId = session.id;  // ← updated here
```

Fork happens inside session-manager, which doesn't have access to `clientState`.

### The same bug exists for edit flow

If edit mode changes the session name/ID (it doesn't currently), or if any session-manager operation creates a new session and joins the client to it, the same stale-routing bug would occur. The fork-from-lobby flow in `index.ts` (`finalizeForkFromLobby`) DOES update clientState correctly — only the in-session fork path (Ctrl+B f) is broken.

## Fix Options

### Option A: Callback pattern (recommended)

Add an `onClientSessionChange` callback to SessionManager, similar to `setOnClientDetach`:

```typescript
// session-manager.ts
private onSessionChangeCallback?: (client: Client, newSessionId: string) => void;

setOnSessionChange(callback: (client: Client, newSessionId: string) => void): void {
  this.onSessionChangeCallback = callback;
}

// In executeFork(), after createSession + leaveSession:
this.onSessionChangeCallback?.(client, forkedSession.id);
```

```typescript
// index.ts
sessionManager.setOnSessionChange((client, newSessionId) => {
  const state = clientState.get(client.id);
  if (state) {
    state.mode = 'session';
    state.sessionId = newSessionId;
  }
});
```

**Pros:** Clean separation — session-manager doesn't know about clientState, just emits an event. Reusable for any future operation that moves a client between sessions.

**Cons:** One more callback to wire up at init time.

### Option B: Return value from executeFork

Have `executeFork` return the new session ID, and handle routing in the caller:

```typescript
// session-manager.ts
executeFork(...): string | null {  // returns new session ID
  // ...
  return forkedSession.id;
}
```

But this doesn't work cleanly because `executeFork` is called from `handleForkInput` which is called from `handleInput` — the return value would need to bubble up through multiple layers.

**Pros:** No new callback wiring.

**Cons:** Leaky — requires handleInput callers to check return values. Fragile.

### Option C: Pass clientState into SessionManager

Give SessionManager direct access to the clientState map:

```typescript
// session-manager.ts constructor
constructor(options: SessionManagerOptions & { clientState: Map<string, ClientState> }) {
```

Then executeFork updates it directly.

**Pros:** Simple, one-line fix in executeFork.

**Cons:** Violates separation of concerns — session-manager shouldn't know about index.ts's state management. Creates a circular dependency risk.

### Option D: Event emitter pattern

Make SessionManager extend EventEmitter:

```typescript
this.emit('client-session-change', { client, sessionId: forkedSession.id });
```

```typescript
// index.ts
sessionManager.on('client-session-change', ({ client, sessionId }) => {
  const state = clientState.get(client.id);
  if (state) { state.mode = 'session'; state.sessionId = sessionId; }
});
```

**Pros:** Standard Node.js pattern, extensible for other events.

**Cons:** Heavier than a simple callback for a single use case.

---

## Recommendation

**Option D (EventEmitter)** — SessionManager is the central coordinator that knows when things happen (sessions created, destroyed, forked, clients moved). Index.ts, API server, and eventually svg-terminal's websocket layer all need to react. This trajectory (2 events today, 5+ likely soon) justifies EventEmitter over ad-hoc callbacks.

Migrate existing `setOnClientDetach` and `onSessionEndCallback` into the EventEmitter at the same time — replaces two ad-hoc callback patterns with one consistent mechanism. Zero new dependencies (Node built-in).

**Events to emit initially:**
- `client-session-change` — client moved between sessions (fixes the fork bug)
- `client-detach` — client detached from session (replaces `setOnClientDetach`)
- `session-end` — session exited (replaces `onSessionEndCallback`)

**Future events (no code needed now, but the pattern supports them):**
- `session-created`, `session-destroyed`, `session-renamed`
- `client-kicked`, `access-changed`, `claude-id-discovered`

Estimated fix: ~40 lines across 2 files (including migration of existing callbacks).
