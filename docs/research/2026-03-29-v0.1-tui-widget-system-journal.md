## The Problem

claude-proxy has 22 interactive UI elements across 4 files. They share 5 fundamental widget types but each is implemented independently:

| Widget Type | Instances | Pattern |
|-------------|-----------|---------|
| List Picker (arrow nav, enter select) | 6 | Lobby, server, restart, workdir, help menu, edit settings menu |
| Checkbox Picker (arrow nav, space toggle) | 5 | Users, groups, admins, export, edit-users/groups/admins |
| Text Input (type, backspace, enter) | 5 | Session name, run-as user, password create, password join, edit password |
| Y/N Prompt (single key) | 4 | Hidden, view-only, public, danger mode |
| Combo (picker + freehand text) | 1 | Working directory |
| Scrollback Viewer (pagination) | 1 | Scrollback viewer |

14 separate arrow-key handlers. 18 separate Math.max/min cursor bounds. Every picker re-implements rendering with inline ANSI escape codes.

### Why This Matters

1. **Web interface** — browser clients need the same creation flow. Can't reuse ANSI-coupled logic.
2. **svg-terminal unification** — svg-terminal needs structured data (items, cursor, selection), not raw ANSI to parse.
3. **OAuth, new flows** — every new picker means copy-pasting another handler.
4. **Bugs propagate** — the wrapping bug (Math.max/min vs modulo) exists in 18 places. Lobby wraps correctly, everything else doesn't.

## Framework Survey

Researched four established TUI frameworks and multi-transport patterns:

### Bubbletea (Go) — Elm Architecture
- **State:** Single immutable model struct
- **Cycle:** `msg → Update() → new model + cmd` then `View() → string`
- **Rendering:** Pure function from model to string (ANSI)
- **Multi-transport:** Not built-in. Charm's Wish library serves Bubbletea over SSH by connecting each session's pty.
- **Focus:** Manual — parent model routes messages to focused child
- **Verdict:** Maximally simple. No magic. But View() returns ANSI strings, so web needs an ANSI-to-HTML converter.

### Ink (React for CLI)
- **State:** Standard React (useState, useReducer)
- **Cycle:** setState → reconcile → render JSX → Yoga flexbox → ANSI output
- **Rendering:** Custom React renderer for terminal. `renderToString()` for non-interactive.
- **Multi-transport:** Partial — renderToString works but not interactive.
- **Focus:** Built-in useFocus/useFocusManager hooks. Tab cycles focusable components.
- **Verdict:** Clean separation via React reconciler pattern, but web transport isn't live.

### Textual (Python) — The Multi-Transport Winner
- **State:** Widget instance attributes + reactive descriptors
- **Cycle:** Event → handler → refresh → compose()/render() + CSS
- **Rendering:** Abstract widget tree + real CSS subset. **Driver abstraction** for output.
- **Multi-transport:** YES. `textual serve` runs same app in web browser. Zero code changes.
- **Focus:** Full DOM-based focus system, event bubbling, key bindings.
- **Verdict:** Most web-like architecture. CSS engine + driver abstraction = true multi-transport.

### Blessed (Node.js) — Classic ncurses-in-JS
- **State:** Mutable widget properties (DOM-like)
- **Cycle:** Mutate → screen.render() → painter's algorithm with damage buffer
- **Rendering:** Manual render() call after state changes
- **Multi-transport:** Terminal only.
- **Focus:** Built-in widget.focus() + event bubbling
- **Verdict:** Mature but imperative, unmaintained, terminal-only. Not a good model for us.

### Ratatui (Rust) — Backend Abstraction
- **State:** App struct + widget traits
- **Rendering:** Widgets write to an abstract Buffer (2D cell grid). Backends convert to terminal/WASM.
- **Multi-transport:** Via pluggable backends — terminal, WASM, etc.
- **Verdict:** Clean backend abstraction. Widget defines cells, backend renders them.

## Key Architectural Patterns

Three patterns for multi-transport TUI:

### 1. Driver/Backend Abstraction (Textual, Ratatui)
Widget definitions produce abstract output. A driver converts to ANSI (SSH), HTML/JSON (web), or spans (svg-terminal). **This is the cleanest path for us.**

### 2. Custom Renderer (Ink)
React reconciler separates "what" from "how". Works but tightly coupled to React. Too heavy for our needs.

### 3. Pure View Functions (Bubbletea)
View() returns a string. Simple but the string is ANSI, so web needs post-processing. Works for SSH-only.

## What claude-proxy Actually Needs

We're not building a general-purpose TUI framework. We need:

1. **5-6 widget types** that handle state + input, produce structured output
2. **A flow engine** that chains widgets into multi-step forms (the creation wizard)
3. **2-3 renderers** that convert structured output to:
   - ANSI escape codes (SSH clients)
   - JSON messages (WebSocket/API for browser clients)
   - Structured screen state (svg-terminal integration)

The Textual model (widget tree + driver abstraction) is the right pattern, simplified for our scope. We don't need CSS, DOM, or flexbox — just widgets that output `{ items, cursor, selected, title, hints }` and renderers that convert that to the right format.

## Open Questions

1. **Should widgets own their rendering at all?** Bubbletea says yes (View returns string). Textual says no (driver does it). For us, widgets returning structured data and letting renderers handle output seems cleaner — especially since we need 3 output formats.

2. **How does the flow engine handle conditional steps?** The creation flow has steps that only appear for admins, or only when not hidden. Conditions need access to the accumulated state from prior steps.

3. **How do existing interactive-menu.ts patterns fit?** The lobby already uses `renderMenu()` and `nextSelectable()` from interactive-menu.ts. These are proto-widgets. The help menu and edit settings menu also use them. The refactor should build on this foundation, not throw it away.

4. **How much of the scrollback viewer fits this model?** It's quite different from pickers/inputs — it's a paginated content viewer with vim-like navigation. May stay as its own thing.

5. **Migration strategy:** Big-bang rewrite vs incremental? The flow engine is the biggest change. Could extract widgets first, then build the flow engine, then migrate one flow at a time.
