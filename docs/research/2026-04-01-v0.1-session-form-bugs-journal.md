# Session Form Bugs Journal v0.1
**Date:** 2026-04-01
**Session:** c9510550-3bf3-4f87-bde0-ca10db1cfbab
**Status:** Published
**Preceding:** none
**Changes from v0.0:** Five bugs found and fixed during live testing of unified session form + fork

## Bug 1: session-form.yaml not in dist/

**Symptom:** Service crash on startup — `ENOENT: no such file or directory, open '/srv/claude-proxy/dist/session-form.yaml'`

**Cause:** TypeScript compiler only copies `.ts` files to `dist/`. The YAML config file stayed in `src/`.

**Fix:** Updated `package.json` build script: `"build": "tsc && cp src/session-form.yaml dist/session-form.yaml"`

**Lesson:** Any non-TypeScript file that code reads at runtime needs explicit copying in the build step. This will happen again if we add more YAML or JSON config files.

## Bug 2: Claude PID discovery fails for sessions with arguments

**Symptom:** Fork fails silently for sessions started with `claude --resume` or any arguments. Returns "Cannot fork: no Claude session ID found."

**Cause:** `findClaudePid()` in `session-store.ts` checked `cmd === 'claude'` but `ps` reports the full command line: `claude --resume`. The string comparison failed.

**Fix:** Split command on space and check the base: `cmd.split(' ')[0] === 'claude'`

**Lesson:** Process command lines from `ps` always include arguments. Never match the full string when looking for a program name. This affects any session started with `--resume`, `--fork-session`, `--dangerously-skip-permissions`, etc.

## Bug 3: Locked fields trap the cursor in edit mode

**Symptom:** Pressing Enter on a locked field (e.g. "Run as user" in fork mode) enters edit mode where the TextInput is locked. All input is ignored except Escape. User appears stuck.

**Cause:** `handleNavigateKey` in FlowEngine created the widget and entered edit mode without checking if the widget was locked.

**Fix:** Check `widget.state?.locked` after creating the widget. If locked, don't enter edit mode — return `{ type: 'none' }`.

**Lesson:** Locked is a display state (show value, indicate not editable) not an interaction state (enter edit and reject input). The form should skip locked fields entirely on Enter.

## Bug 4: Cursor doesn't auto-advance after confirming a field

**Symptom:** User edits session name, presses Enter to confirm, name shows ✓ but cursor stays on the same field. User must manually press Down to advance.

**Cause:** `handleEditKey` in FlowEngine returned to navigate mode after confirming but didn't move the cursor to the next visible field.

**Fix:** After `completedSteps.add(this.currentIndex)`, call `findNextVisible(this.currentIndex, 1)` to auto-advance.

**Lesson:** Form UX expects Enter to both confirm AND advance. Splitting confirm from advance creates a "stuck" feeling even though the system is working correctly.

## Bug 5: Pending fields show [default] instead of pre-filled values

**Symptom:** Fork form shows `[default]` for all fields below the cursor, even though the values are known (pre-filled from source session).

**Cause:** Two issues:
1. `getFlowSummary()` didn't include `value` for pending steps — only for completed steps.
2. FlowEngine's `accumulated` state (used by `displayValue`) didn't include prefill values. The `initialState` passed to FlowEngine only had `_isAdmin`, `_hasRemotes`, `_defaultUser` — not the session data.

**Fix:**
1. `getFlowSummary()` now calls `displayValue` for pending steps too, returns value if available.
2. Fork and edit flows now merge `prefill` into `initialState` via spread: `{ ...prefill, _isAdmin, ... }`.

**Lesson:** If a form has pre-filled values, they must be in the FlowEngine's accumulated state from the start — not just in the widget's `initial` option. The widgets use `initial` for their UI, but `displayValue` reads from `accumulated`. Both paths must have the data.

## Bug 6: Fork/edit used stale cached session ID

**Symptom:** Not directly observed, but the fork code was reading `loadSessionMeta().claudeSessionId` (cached from last backfill) instead of discovering live from the running process.

**Cause:** Backfill runs at 3s/5s/10s after session creation and may not have run yet, or the session could have been resumed with a new Claude session ID since the last backfill.

**Fix:** Fork and edit now call `discoverClaudeSessionId()` first (live from process tree), fall back to stored metadata only if live discovery fails. If live ID differs from stored, the old ID is rotated to `pastClaudeSessionIds[]`.

**Lesson:** Always prefer live state over cached state for mutable data. The PID → `~/.claude/sessions/<pid>.json` path gives real-time truth. Stored metadata is a fallback for dead sessions where the process no longer exists.
