# Unix Socket Transport & Layered OAuth — Stepped v02

**Date:** 2026-04-03  
**Status:** In Progress — for review before planning  
**Preceding:** `v01`  
**Changes from v01:** First-principles redesign. Operations layer extracted from HTTP handlers. Unix socket uses direct JSON-RPC (not HTTP-over-socket). HTTP becomes one adapter, not the core. Deprecation and rework sections updated to reflect the larger refactor.

---

## 1. What we're doing

Separating **what claude-proxy can do** (operations) from **how you talk to it** (transport).

Today these are tangled: business logic lives inside HTTP route handlers in `api-server.ts`. That means every new transport (Unix socket, future IPC, tests) must go through HTTP semantics — or duplicate the logic.

The change:

```
BEFORE (today):
  HTTP route handler → business logic → SessionManager

AFTER:
  ┌──────────────────────────────┐
  │     Operations module        │
  │  listSessions(user)          │
  │  createSession(req)          │
  │  getScreen(sessionId)        │
  │  authStart(provider, cb)     │
  │  authExchange(provider, code)│
  │  ...                         │
  └──────┬──────────┬────────────┘
         │          │
  ┌──────▼────┐  ┌──▼───────────┐
  │ Unix sock │  │ HTTP adapter │
  │ JSON msgs │  │ /api/* routes│
  │ (internal)│  │ (browsers)   │
  └───────────┘  └──────────────┘
```

**Operations** are plain TypeScript functions/methods with typed inputs and outputs. No `req`, no `res`, no HTTP status codes. They validate, execute, and return results or throw.

**Adapters** are thin wrappers that translate a transport's wire format into an operations call and translate the result back.

## 2. The operations module

### What moves out of `api-server.ts`

Every block that currently does `if (url.pathname === '/api/...')` contains two things mixed together: (a) HTTP parsing/response and (b) actual work. The actual work becomes an operation.

| Operation | Current location in `api-server.ts` | Input | Output |
|-----------|-------------------------------------|-------|--------|
| `listSessions(user)` | `GET /api/sessions` (~line 633) | username | `Session[]` (filtered by access) |
| `createSession(req)` | `POST /api/sessions` (~line 349) | `{ name, launchProfile, runAsUser, workDir, access, ... }` | `{ id, name }` |
| `getDeadSessions(user)` | `GET /api/sessions/dead` (~line 392) | username | `DeadSession[]` |
| `destroySession(id, user)` | `DELETE /api/sessions/:id` (~line 411) | sessionId, username | `void` (throws if not allowed) |
| `getSessionSettings(id, user)` | `GET /api/sessions/:id/settings` (~line 426) | sessionId, username | settings object |
| `updateSessionSettings(id, user, patch)` | `PATCH /api/sessions/:id/settings` (~line 445) | sessionId, username, partial settings | `void` |
| `restartSession(id, user, settings)` | `POST /api/sessions/:id/restart` (~line 473) | sessionId, username, optional overrides | `{ id, name }` |
| `forkSession(id, user, settings)` | `POST /api/sessions/:id/fork` (~line 533) | sessionId, username, fork options | `{ id, name }` |
| `getScreen(id, user)` | `GET /api/session/:id/screen` (~line 663) | sessionId, username | screen state object |

| `authStart(provider, callbackUrl)` | new | provider name, callback URL | `{ authUrl, state }` |
| `authExchange(provider, code, state)` | new | provider, auth code, state | `{ identity, token }` |
| `authMe(token)` | `GET /api/auth/me` (~line 312) | signed token | `{ linuxUser, displayName }` |

### What the operations module looks like

A class or namespace that holds `SessionManager`, `UserStore`, `Provisioner`, `OAuthManager`, etc. as injected dependencies. Each method is a pure operation — validate input, call dependencies, return typed result or throw.

```typescript
// src/operations.ts (name TBD)
export class ProxyOperations {
  constructor(
    private sessionManager: SessionManager,
    private config: Config,
    private dirScanner: DirScanner,
    private userStore?: UserStore,
    private provisioner?: Provisioner,
    private oauthManager?: OAuthManager,
    private githubAdapter?: GitHubAdapter,
    private cookieSecret?: string,
  ) {}

  listSessions(username: string): SessionInfo[] { ... }
  createSession(req: CreateSessionRequest): { id: string; name: string } { ... }
  getScreen(sessionId: string, username: string): ScreenState { ... }
  authStart(provider: string, callbackUrl: string): { authUrl: string; state: string } { ... }
  authExchange(provider: string, code: string, state: string): { identity: UserIdentity; token: string } { ... }
  // ...
}
```

### Request/response types

Shared TypeScript types for inputs and outputs. These types become the **contract** — both adapters (HTTP and socket) reference them. They are also what the launch-profiles `CreateSessionRequest` from the add-terminal research was pointing to.

## 3. The Unix socket adapter

### Protocol: JSON messages on a persistent connection

No HTTP. No URLs. No status codes. Just newline-delimited JSON over a Unix domain socket:

**Request:**
```json
{ "id": 1, "method": "listSessions", "params": { "user": "greg" } }
```

**Response:**
```json
{ "id": 1, "result": [{ "id": "cp-FOO", "name": "FOO", ... }] }
```

**Error:**
```json
{ "id": 1, "error": { "code": "NOT_FOUND", "message": "Session not found" } }
```

**Streaming (terminal output):** After a `subscribe` call, the server pushes tagged messages on the same connection:

```json
{ "id": 2, "method": "subscribe", "params": { "sessionId": "cp-FOO" } }
// → server pushes:
{ "event": "screen", "sessionId": "cp-FOO", "data": { "lines": [...], "cursor": {...} } }
{ "event": "delta", "sessionId": "cp-FOO", "data": { "changed": {...} } }
```

**Input (keystrokes from dashboard):**
```json
{ "method": "input", "params": { "sessionId": "cp-FOO", "data": "ls\r", "user": "greg" } }
```

This is a lightweight JSON-RPC-like protocol. No third-party dependency — just `net.createServer()`, line-split, `JSON.parse`, dispatch to operations, `JSON.stringify` the result.

### Socket location and permissions

| Item | Value |
|------|-------|
| Path | `/run/claude-proxy/api.sock` |
| Owner | `root` |
| Group | `cp-services` |
| Mode | `0660` |
| svg-terminal service user | member of `cp-services` |

Same UGO model as tmux session sockets. Managed by `SocketManager` or startup script.

## 4. The HTTP adapter

### What it becomes

A thin translation layer. Each route:

1. Parses the HTTP request (URL, body, cookies, headers)
2. Extracts the authenticated user (from cookie)
3. Calls the corresponding operation
4. Serializes the result to JSON with an HTTP status code

The HTTP adapter **only** exists for:
- **Browsers** (direct access in standalone mode, or if someone needs it)
- **Dev/debug** (`curl` convenience)
- **OAuth redirects** (browsers must follow HTTP redirects from providers)

It does **not** exist as the primary internal transport.

### What stays in `api-server.ts`

- HTTP parsing, CORS, static file serving, cookie handling
- WebSocket upgrade for browser terminal streaming
- OAuth redirect routes (`GET /api/auth/login`, `GET /api/auth/callback`) — these are inherently HTTP (browser redirects)

### What leaves `api-server.ts`

All business logic. The `if (url.pathname === '/api/sessions')` blocks become:

```typescript
// Before (today): ~30 lines of logic inline
if (url.pathname === '/api/sessions' && req.method === 'GET') {
  const sessions = sessionManager.listSessions().map(s => { /* ... */ });
  res.writeHead(200, { 'Content-Type': 'application/json' });
  res.end(JSON.stringify(sessions));
}

// After: ~5 lines of adapter
if (url.pathname === '/api/sessions' && req.method === 'GET') {
  const user = requireAuth(req, res); if (!user) return;
  const sessions = ops.listSessions(user.linuxUser);
  json(res, 200, sessions);
}
```

## 5. What this changes vs v01

| Item | v01 (pragmatic) | v02 (first principles) |
|------|-----------------|----------------------|
| Internal transport | HTTP over Unix socket (`socketFetch`) | JSON messages over Unix socket (no HTTP) |
| Operations | Inline in `api-server.ts` handlers | Extracted to `src/operations.ts` |
| HTTP adapter | **Is** the API | Thin wrapper that calls operations |
| Socket adapter | N/A (was just HTTP piped through socket) | New: `src/socket-server.ts` — JSON-RPC dispatcher |
| svg-terminal client | `socketFetch()` wrapper mimicking `fetch()` | Socket client sending JSON messages directly |
| `api-server.ts` size | ~960 lines (stays same) | **Shrinks** — logic moves to operations |
| New code | ~60 lines (internal auth endpoints) | ~200-300 lines (`operations.ts` + `socket-server.ts`) but most is extracted, not new logic |
| Rework | Minimal | **Medium** — refactor `api-server.ts` to extract logic (not rewrite; move and call) |

## 6. What gets deprecated or removed

| Item | After this change |
|------|-------------------|
| **TCP `127.0.0.1:3101` as primary internal transport** | Replaced by Unix socket. TCP optional for dev/browsers. |
| **Business logic inline in `api-server.ts`** | Extracted to operations module. Route handlers become adapters. |
| **`CLAUDE_PROXY_API` constant in `server.mjs`** | Replaced by socket path. svg-terminal talks JSON-RPC, not HTTP. |
| **Direct `fetch()` to claude-proxy in `server.mjs`** | Replaced by socket client (persistent connection, JSON messages). |
| **Direct `new WebSocket()` to claude-proxy in `server.mjs`** | Replaced by `subscribe` method on socket connection. |
| **`web/index.html` stub** | Replaced with minimal standalone login page (or redirect). |
| **OAuth wiring spec (TCP-focused sections)** | `public_base_url` concept changes — svg-terminal owns the public URL. Revision note on spec. |

## 8. Layered OAuth (unchanged from v01)

Same three-layer model:

| Layer | What |
|-------|------|
| **0** | claude-proxy operations module owns all auth logic |
| **1** | HTTP adapter serves standalone login pages (CSS-classed, themeable) |
| **2** | svg-terminal calls `authStart` / `authExchange` operations via socket; renders its own themed pages |

The operations `authStart` and `authExchange` are transport-agnostic. Both the HTTP adapter and socket adapter can call them.

## 10. Rework assessment

| Category | Effort | Description |
|----------|--------|-------------|
| **Extract** operations from `api-server.ts` | Medium | ~15 route handlers → ~15 operations methods. Logic moves; HTTP wrappers stay. Mechanical refactor. |
| **New** `socket-server.ts` | Medium | JSON-RPC dispatcher over Unix socket. ~100-150 lines. Method table maps to operations. |
| **New** `operations.ts` | Medium | Class with typed methods. Most code extracted from `api-server.ts`, not written from scratch. |
| **New** request/response types | Small | TypeScript interfaces for operation inputs/outputs. |
| **Refactor** `api-server.ts` | Medium | Handlers shrink to 3-5 lines each (parse → call operation → respond). Total file shrinks significantly. |
| **New** svg-terminal socket client | Medium | Persistent socket connection, JSON message send/receive, subscribe for streams. Replaces 9 `fetch`/`WebSocket` call sites. |
| **Refactor** `server.mjs` call sites | Small | Replace `fetch()`/`WebSocket()` with socket client methods. |
| **New** standalone login HTML | Small | ~50 lines, CSS classes. |
| **Config** extension | Small | `api.socket`, `api.socket_mode`, `api.socket_group` in types/YAML. |
| **Tests** | Medium | Operations are easier to test than HTTP routes (no mock request/response). Some existing API tests may need updates. |

**Total estimate:** Larger than v01 (~9 hours) — closer to **16-20 hours** of implementation. But the result is a cleaner architecture that the add-terminal research and ROADMAP already assumed would exist.

## 11. Open questions

1. **Operations module naming:** `src/operations.ts` or `src/proxy-api.ts` or `src/service.ts`? Should it be a class or a bag of functions with injected deps?

2. **Socket protocol:** JSON-RPC 2.0 (formal spec, client libraries exist) or simplified JSON messages (less ceremony, we control both sides)?

3. **Streaming model:** One persistent socket connection per svg-terminal process with multiplexed subscriptions? Or one connection per terminal stream?

4. **Does svg-terminal need to be refactored at the same time** or can claude-proxy ship the operations + socket adapter first, then svg-terminal migrates from HTTP to socket in a later step? (TCP adapter stays as bridge during migration.)

5. **Should the HTTP adapter be in the same process** (as today) or a separate thin proxy process? Same process is simpler; separate process is more isolated.

6. **Test strategy:** Test operations directly (unit tests, no transport) + integration tests through each adapter?
