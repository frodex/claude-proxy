## Summary

Directory selection logic is scattered across 6 files with no security model. Discovered during fork testing — the fork form only showed `~ (home directory)` because `getDirItems` context wasn't passed to session-manager's fork/edit flows.

Analysis identified 5 security concerns (filesystem visibility, remote hosts, user permissions, global history, mkdir-p), 7 design questions (run-as-user model, config, per-user history, remote completion, widget vs service, reusability, mkdir responsibility), and 4 integration options.

Recommendation: a `DirectorySelector` service that owns security, scanning, history (per-user), and completion. Called directly by any flow. Solves both the plumbing gap and the security gap.
