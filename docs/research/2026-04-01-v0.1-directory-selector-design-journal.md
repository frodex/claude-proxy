## The Problem

Directory selection for session creation/editing is currently scattered across multiple files with no security model. The logic is split between:

| Piece | File | What it does |
|-------|------|-------------|
| Filesystem scanning | `src/dir-scanner.ts` | Walks roots, finds git repos, tracks MRU history |
| Tab-completion | `src/index.ts` (`tabComplete()`) | Reads local filesystem, completes partial paths |
| List assembly | `src/index.ts` (`getDirItems` callback) | Combines home + history + scanned into picker items |
| Validation | Was inline in old creation flow | `existsSync` check, `mkdir -p` on launch |
| Remote awareness | Form rendering logic | No tab-complete for remote, no scan for remote |

This creates three problems:
1. Logic is duplicated/scattered — fork and edit flows can't access the directory list because `dirScanner` lives in `index.ts`
2. No security model — root sees everything, no filtering by user
3. No clear ownership — who decides what directories a user can see?

## Security Concerns

### 1. Directory visibility reveals filesystem structure
Scanning `/srv`, `/home`, `/root` exposes what projects and users exist on the system. A non-root user shouldn't see root's directories or other users' home directories.

### 2. Remote hosts have different filesystems
The local scan results don't apply to ct100 or other remotes. Currently remote sessions get no directory suggestions — just freehand typing. But tab-completion attempts to read the LOCAL filesystem even for remote sessions (bug: the remote flag disables it in the form, but the underlying function doesn't know about remotes).

### 3. User permission boundaries
A user running `su - greg` should see greg's directories, not root's. The current scanner runs as root and returns everything. There's no filtering by the target user's permissions.

### 4. History is global
`data/dir-history.json` is shared across all users. User A's history shows up for User B. Session working directories should be per-user.

### 5. New directory creation
The proxy runs `mkdir -p` as root (or via `su` as the target user) to create directories that don't exist. This means a user can create arbitrary directories on the filesystem through the session creation form.

## What a Directory Selector Should Own

A self-contained `DirectorySelector` that encapsulates:

| Responsibility | Current owner | Should be |
|---------------|--------------|-----------|
| What directories to scan | DirScanner class | DirectorySelector, scoped to user |
| Scan root paths | Hardcoded `/srv`, `/home`, `/root` | Configurable per user/role, from YAML |
| History tracking | Global `dir-history.json` | Per-user history |
| Tab-completion | `tabComplete()` in index.ts | DirectorySelector, local or remote |
| Remote tab-completion | Not implemented | SSH to remote, `ls` partial path |
| Validation | Inline existsSync | DirectorySelector, as user (not root) |
| Permission filtering | Not implemented | Only show dirs user can access |
| New directory creation | `mkdir -p` in pty-multiplexer | DirectorySelector validates, tmux launch creates |
| List assembly | `getDirItems` callback | DirectorySelector.getItems() |

## Design Questions (Open)

### 1. Who does the DirectorySelector run as?
- **As root, filtering results?** Simple but trusts the filter logic. A bug exposes everything.
- **As the target user (su)?** Correct by construction — only sees what the user can see. But slower (su + ls for every completion) and requires the user account to exist.
- **Hybrid?** Scan as root at startup (cached), filter by user permissions at request time. Fast but the filter is a second security layer, not the primary one.

### 2. Where does the configuration live?
- Scan roots per user/role? e.g., students see `/home/<user>/projects`, admins see `/srv`
- YAML in `claude-proxy.yaml`?
- Per-user config in the user store?

### 3. Should history be per-user?
Current: one global `dir-history.json`. If yes, where? `data/dir-history/<username>.json`? In the SQLite user store?

### 4. Remote tab-completion — worth building?
Current: disabled for remote sessions. Could SSH to the remote and `ls` the partial path. Adds latency (SSH round-trip per Tab press). Worth it?

### 5. Should this be a widget or a data service?
- **Widget:** `DirectorySelector` extends ComboInput, owns its own data fetching and rendering. Self-contained.
- **Data service:** `DirectoryService` provides data, a plain ComboInput renders it. Separation of concerns.
- **Both:** `DirectoryService` for data, `DirectorySelector` widget composes `ComboInput` + `DirectoryService`.

### 6. Reusability
Will this be used outside session forms? A file browser? A project picker? If so, the data service approach is better — multiple UIs can consume the same service.

### 7. mkdir -p responsibility
Currently the tmux launch command includes `mkdir -p` for the working directory. Should the DirectorySelector validate and create during selection, or should creation stay in the launch path? Creating during selection confirms the path works. Creating at launch is simpler but errors are late.

## Current State Summary

```
User presses Enter on "Working directory" in session form
  ↓
ComboInput created with items from getDirItems callback
  ↓
getDirItems calls dirScanner.getHistory() + dirScanner.scan()
  ↓
DirScanner walks /srv, /home, /root for .git dirs (as root)
  ↓
History comes from global dir-history.json
  ↓
User sees list, picks or types freehand
  ↓
Tab-completion reads local filesystem (as root, even for remote sessions)
  ↓
Selected path stored in session metadata
  ↓
tmux launch: mkdir -p <path> && cd <path> && claude
```

No user filtering at any step. No per-user history. No remote completion. No permission validation.

## Example Case: The Integration Problem That Surfaced This

`getDirItems` is a callback function that gets passed through as context — it's not duplicated. There's one implementation in `index.ts` (lines 162-172) that calls `dirScanner.getHistory()` and `dirScanner.scan()`. It's injected into the session form via the `WidgetContext` interface.

**The problem:** `index.ts` creates the context with `getDirItems` for the creation flow, but `session-manager.ts` creates the context for fork/edit flows WITHOUT `getDirItems` because it doesn't have access to `dirScanner`.

This is why the fork form only shows `~ (home directory)` — the other items require `getDirItems` which requires `dirScanner` which lives in `index.ts`.

**Three options considered:**

1. **Pass dirScanner to SessionManager** as a constructor option — then it builds its own `getDirItems`. Simple but creates a dependency from session-manager → dir-scanner. Session-manager currently knows nothing about directories.

2. **Pass the getDirItems function** to SessionManager as a constructor option — decoupled from dirScanner. Session-manager doesn't know how the list is built, just that it can call a function to get it. But this is still a "pass the whole world through the constructor" pattern.

3. **Move the fork/edit flow initiation to index.ts** where dirScanner already lives — session-manager just handles the session lifecycle, index.ts handles ALL form flows. Clean separation: session-manager = session operations, index.ts = user interaction flows. But the fork/edit flows are triggered from session-manager's help menu (Ctrl+B e/f), so they'd need to callback to index.ts.

**Why none of these are right yet:**

All three options assume the current directory scanning model is correct — root scans everything, no filtering, global history, no user awareness. Whichever plumbing option we pick, we're wiring up an insecure data source. The right answer depends on the security model decided in the design questions above. If the DirectorySelector becomes a proper service that runs as the target user, the integration question changes entirely:

4. **DirectorySelector service** — a standalone module that takes a username, returns directories that user can see. Session-manager and index.ts both call it directly. No callback passing, no constructor injection. The service owns the security model, the scanning, the history (per-user), and the completion. The form just asks it for data.

This is option 5 from the design questions (data service) applied to the integration problem. It solves the plumbing AND the security in one design.
