## 1. Context

Work spanned two repositories:

- **claude-proxy:** Add a **Unix domain socket** listener that speaks **newline-delimited JSON-RPC** to the same `ProxyOperations` surface used by the HTTP/WebSocket API on port 3101. Motivation: local integrations (notably **svg-terminal**) should not depend on TCP to `127.0.0.1:3101` for session discovery, screen data, and input; a socket under `/run/` is easier to permission, firewall, and reason about than another listening port for purely local traffic.
- **svg-terminal:** Replace `fetch()` / `WebSocket` call sites that targeted claude-proxy’s HTTP API with a **single long-lived `net.createConnection`** client that multiplexes requests and `terminal` events.

After implementation, the host still ran svg-terminal via **ad hoc** `node server.mjs` / `nohup` / `restart-server.sh`. That is fragile across reboots and duplicates process supervision that **systemd** already provides for claude-proxy.

---

## 2. Design choices

### 2.1 Keep HTTP + WebSocket on 3101 alongside the socket

We did **not** remove `api-server.ts`. Browsers and OAuth redirects still need HTTP; the socket is **additive**. Config in `claude-proxy.yaml` uses an `api:` block with both `port: 3101` and `socket: /run/claude-proxy/api.sock`.

### 2.2 JSON-RPC shape and events

Requests: `{ id, method, params }`. Responses: `{ id, result | error }`. Streaming uses `{ event, sessionId, data }` where `event === 'terminal'` and `data` matches the existing WebSocket payload shape (`type: 'screen' | 'delta'`, etc.).

Methods were aligned to **real** `ProxyOperations` names (`getSessionScreen`, `startAuth`, …), not an older draft list in a plan document.

### 2.3 Resize and scroll on the socket

The dashboard sends `resize` and `scroll` messages for claude-proxy sessions (previously only over the WebSocket to `:3101`). The socket server gained **`resize`** and **`scroll`** RPC methods that mirror the WebSocket handler logic so svg-terminal can stay socket-only for those paths.

### 2.4 svg-terminal environment

- `CLAUDE_PROXY_SOCKET` — defaults to `/run/claude-proxy/api.sock`.
- `CLAUDE_PROXY_USER` — defaults to `root`; used where there is no authenticated browser user (e.g. legacy `/api/pane`).

### 2.5 systemd unit for svg-terminal

**Why:** Persist across reboots; single supervised process; logs in `journalctl`; clear dependency order vs claude-proxy.

**Ordering:** `After=network.target claude-proxy.service` so the claude-proxy Unix socket is typically created before svg-terminal starts. svg-terminal’s client reconnects on failure, but ordering reduces race noise at boot.

**Why not fold svg-terminal into claude-proxy:** Different release cadence, different repo, and svg-terminal can run tmux-only without claude-proxy; keeping a separate unit preserves that separation.

**Unit file location:** `/etc/systemd/system/svg-terminal.service` (not committed to git — lives on the host). The PRD amendment duplicates the substantive contents for spec readers.

**Port:** `PORT=3200` via `Environment=` in the unit (override with `systemctl edit` if needed).

---

## 3. What we rejected or deferred

- **socket-only claude-proxy (no TCP)** for production: possible via YAML (`api` with `socket` but no `port`), not chosen for this host so browsers and `curl` debugging on 3101 remain available.
- **Committing the systemd unit inside the svg-terminal repo:** deferred; could add `deploy/svg-terminal.service.example` later for Ansible/gitops. For now the amendment + journal are the portable record.
- **tmux session termination on proxy restart:** explicitly out of scope; `KillMode=process` on claude-proxy predates this work; svg-terminal restart does not invoke `tmux kill-server`.
