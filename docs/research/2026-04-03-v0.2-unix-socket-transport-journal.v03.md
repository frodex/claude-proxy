## 1. What we're doing

Separating **what claude-proxy can do** (operations) from **how you talk to it** (transport), while **fixing security findings** in every module we touch.

```
BEFORE (today):
  HTTP route handler → business logic (inline) → SessionManager
  Auth modules built but not wired
  execSync with string interpolation everywhere
  No authentication on any API route

AFTER:
  ┌──────────────────────────────────┐
  │     Operations module            │
  │  (typed inputs, validated,       │
  │   auth-gated, execFileSync)      │
  └──────┬──────────┬────────────────┘
         │          │
  ┌──────▼────┐  ┌──▼───────────┐
  │ Unix sock │  │ HTTP adapter │
  │ JSON msgs │  │ /api/* routes│
  │ (internal)│  │ (browsers)   │
  └───────────┘  └──────────────┘
```

## 3. The Unix socket adapter

JSON messages on a persistent connection. No HTTP.

**Request:** `{ "id": 1, "method": "listSessions", "params": { "user": "greg" } }`  
**Response:** `{ "id": 1, "result": [...] }`  
**Error:** `{ "id": 1, "error": { "code": "NOT_FOUND", "message": "..." } }`  
**Streaming:** After `subscribe`, server pushes `{ "event": "screen", "sessionId": "...", "data": {...} }`

Socket at `/run/claude-proxy/api.sock`, owner `root`, group `cp-services`, mode `0660`.

---

## 4. The HTTP adapter

Thin wrapper. Each route: parse HTTP → call operation → serialize response. Stays in `api-server.ts` but shrinks to ~5 lines per route.

Exists for: browsers (standalone mode), dev/debug (`curl`), OAuth redirects (inherently HTTP).

---

## 5. Layered OAuth

| Layer | What |
|-------|------|
| **0** | Operations module owns all auth logic (`OAuthManager`, `resolveUser`, token signing) |
| **1** | HTTP adapter serves standalone login pages (CSS-classed, themeable) |
| **2** | svg-terminal calls `authStart`/`authExchange` operations via socket; renders its own themed pages |

Two internal operations (not HTTP routes):
- `authStart(provider, callbackUrl)` → `{ authUrl, state }`
- `authExchange(provider, code, state)` → `{ identity, token }`

## 7. What gets deprecated

| Item | Replacement |
|------|-------------|
| TCP `127.0.0.1:3101` as primary | Unix socket (TCP optional for dev) |
| Business logic inline in `api-server.ts` | Operations module |
| `CLAUDE_PROXY_API` / `fetch()` in `server.mjs` | Socket client with JSON messages |
| `web/index.html` stub | Minimal login page or redirect |
| `execSync` with interpolation (throughout) | `execFileSync` with array args |
| Wildcard CORS `*` | Configured allowed origins |
| Hardcoded `'default-insecure-secret'` | Refuse startup without real secret |
| SHA-256 password hashing (no salt) | `crypto.scrypt` with random salt |
| Auto-provision any OAuth user | Approval gate (config: `auto_provision`) |
| Email-based account linking | Provider link match or admin approval |

## 12. Security findings addressed by this work

Findings from `/root/security-audit-findings.csv` that fall within modules we're modifying. Organized by the refactor task that fixes each one.

### 12.1 Fixed by operations extraction + HTTP adapter hardening (`api-server.ts`)

| Line(s) | Severity | Finding | How it's fixed |
|---------|----------|---------|----------------|
| 184-186 | **HIGH** | Wildcard CORS `*` | HTTP adapter sets configured `Access-Control-Allow-Origin` (not `*`) |
| 172-178 | **HIGH** | No body size limit | HTTP adapter adds `Content-Length` check (e.g. 1MB max) before `readBody` |
| 290 | **HIGH** | Hardcoded `'default-insecure-secret'` fallback | Operations module refuses to start if `cookieSecret` is undefined when auth is required |
| 300-301 | Medium | Cookie missing `Secure` flag | HTTP adapter sets `Secure` when `config.api.tls` or behind reverse proxy |
| 349-388 | **HIGH** | `POST /api/sessions` unauthenticated | HTTP adapter calls `requireAuth()` before `ops.createSession()` |
| 367-368 | **HIGH** | `fakeClient.username = 'root'` | Operations uses authenticated `user.linuxUser`, not hardcoded root |
| 377 | **HIGH** | `dangerousSkipPermissions` from untrusted input | Operations validates admin-only before accepting the flag |
| 411-423 | **HIGH** | `DELETE` unauthenticated | HTTP adapter requires auth; operations checks `canUserEditSession` |
| 444-469 | **HIGH** | `PATCH settings` unauthenticated | Same pattern — auth + ACL check |
| 472-530 | **HIGH** | `POST restart` unauthenticated | Same |
| 532-595 | **HIGH** | `POST fork` unauthenticated | Same |
| 607-616 | **HIGH** | `GET /api/users` exposes user list | Auth required; optionally scope to admin-only |
| 619-630 | Medium | `GET /api/groups` exposed | Auth required |
| 600-603 | Medium | `GET /api/remotes` exposed | Auth required |
| 633-652 | **HIGH** | `GET /api/sessions` unfiltered | Operations uses `listSessionsForUser(user)` |
| 730-738 | **HIGH** | WebSocket no auth | Validate cookie on upgrade; close `4401` if invalid |
| 849-867 | **HIGH** | WebSocket input no auth | Same — reject unauthenticated streams |
| 892-903 | Medium | Command injection in capture-pane | Use `execFileSync` with array args |
| 238 | Medium | Verbose OAuth error to client | HTTP adapter returns generic error; logs detail server-side |

### 12.2 Fixed by auth module wiring + hardening (`src/auth/*`)

| File | Severity | Finding | How it's fixed |
|------|----------|---------|----------------|
| `provisioner.ts:9-10` | **HIGH** | `execSync` with interpolated `displayName` → injection | `execFileSync('useradd', ['-m', '-c', displayName, username])` — no shell |
| `provisioner.ts:9-86` | **HIGH** | All commands use `execSync` (shell=true) | Replace all with `execFileSync` (no shell) |
| `provisioner.ts:22-24` | **HIGH** | `curl \| bash` without integrity check | Document as known risk; add `--fail` + hash check where feasible |
| `provisioner.ts:17` | Medium | `deleteSystemAccount` unsanitized username | Validate `^[a-z_][a-z0-9_-]{0,31}$` before any system call |
| `provisioner.ts:28-29` | Medium | `addToGroup` etc. unsanitized group | Same validation regex |
| `resolve-user.ts:61-81` | **HIGH** | Auto-provision without approval | Gate on `config.auth.auto_provision`; default `false`; new users go to `pending` state |
| `resolve-user.ts:53-58` | **HIGH** | Email-based linking without verification | Require matching provider link, not just email match; or require admin approval for first-time link |
| `github-adapter.ts:49-50` | Medium | Unverified email fallback | Throw if no `primary && verified` email found |
| `oauth.ts:38-43` | Medium | No PKCE | Add `code_challenge`/`code_verifier` using `openid-client` PKCE support |
| `session-cookie.ts:14-20` | Medium | Cleartext cookie payload | Option: encrypt payload with `aes-256-gcm` before base64url encoding |
| `session-cookie.ts:23-46` | Medium | No server-side revocation | Add token nonce + revocation set (in-memory or SQLite) |
| `session-cookie.ts:41` | Medium | `exp && ...` bypasses `exp=0` | Change to `payload.exp !== undefined && payload.exp !== null` |
| `middleware.ts:17-43` | Medium | No CSRF | Add `X-CSRF-Token` header check on state-changing requests (`POST`/`PATCH`/`DELETE`) |
| `user-store.ts:34` | Medium | Encryption key SQL-injected via pragma | Use parameterized approach: `db.pragma(\`key=?\`, [encryptionKey])` or escape properly |
| `user-store.ts:31-37` | Medium | No file permissions on DB | `fs.chmodSync(dbPath, 0o600)` after creation |
| `user-store.ts:31-37` | Medium | Encryption optional | Log warning if `encryptionKey` absent; require in production config |
