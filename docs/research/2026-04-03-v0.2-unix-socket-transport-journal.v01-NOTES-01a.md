## 1. What we're considering

Two related changes:

**A) Unix domain socket** as the primary transport between claude-proxy's API server and svg-terminal (replacing TCP `127.0.0.1:3101`).

**B) Layered OAuth** where claude-proxy owns all auth logic and works standalone, and svg-terminal hooks into it via internal API endpoints over that socket.

These are separable — you could do B without A (keep TCP) or A without B (socket but same auth model). But they complement each other and are presented together.

## 2. What exists today and what it means

### claude-proxy API server (`src/api-server.ts`)

**Today:** `http.createServer()` bound to TCP `127.0.0.1:3101`. Serves:
- `/api/sessions`, `/api/session/:id/screen`, `/api/session/:id/stream` (WebSocket), plus auth routes, settings, restart, fork, directories, users, groups
- Static files from `web/` (one placeholder `index.html`)
- WebSocket upgrade for terminal streaming

**Callers:**
- svg-terminal `server.mjs` — 7 call sites using `CLAUDE_PROXY_API = 'http://127.0.0.1:3101'` for HTTP fetch + 2 sites using `ws://127.0.0.1:3101` for WebSocket

**If we change to Unix socket:** The HTTP server itself doesn't change (same routes, same handlers). Only the `server.listen()` call changes from `listen(port, host)` to `listen(socketPath)`. Node.js handles this natively — no library changes.

### svg-terminal connections (`server.mjs`)

**Today:** 7 `fetch()` calls and 2 `new WebSocket()` calls to hardcoded `127.0.0.1:3101`.

| Call site | What it does | Line |
|-----------|-------------|------|
| `fetch(CLAUDE_PROXY_API + '/api/sessions')` | Session discovery (2 places) | 357, 613 |
| `fetch(CLAUDE_PROXY_API + '/api/session/:id/screen')` | Initial screen fetch (2 places) | 231, 668 |
| `new WebSocket('ws://127.0.0.1:3101/api/session/:id/stream')` | Terminal stream bridge (2 places) | 544, 1477 |
| `fetch(CLAUDE_PROXY_API + '/api/session/:id/screen')` | `/api/pane` proxy for socket sessions | 231 |

**If we change to Unix socket:**
- `fetch()` → **cannot use `fetch()` with Unix sockets** in Node.js. Must switch to `http.request({ socketPath, path })` or use a thin wrapper. This is ~15 lines of utility code, not a library.
- `new WebSocket()` → the `ws` library (v8.20.0, installed in both repos) supports Unix sockets via the `createConnection` option or the `agent` option. Requires a small change per call site.

### OAuth auth modules (`src/auth/`)

**Today:** 6 files, all built and tested but **not wired** into `startApiServer()`:

| File | Lines | What | Status |
|------|-------|------|--------|
| `oauth.ts` | ~90 | `OAuthManager` — Google/Microsoft OIDC via `openid-client` | Built, not called from `index.ts` |
| `github-adapter.ts` | ~60 | `GitHubAdapter` — GitHub OAuth (non-OIDC) | Built, not called |
| `resolve-user.ts` | ~93 | `resolveUser()` — maps OAuth/SSH identity → Linux user, provisions if new | Built, not called from API |
| `user-store.ts` | ~120 | `SQLiteUserStore` — users table + provider links | Built, not called |
| `session-cookie.ts` | ~45 | `createSessionCookie` / `validateSessionCookie` — HMAC-SHA256 signed | Built, not called from API routes |
| `middleware.ts` | ~40 | `createAuthMiddleware` — validates cookie, sets `req.user` | Built, **not applied** to any routes |

**Total:** ~450 lines of auth infrastructure sitting unused.

**If we add layered OAuth:** These modules get **wired** (finally), plus two new internal endpoints (~60 lines). No rewrite — they were designed for this.

### `web/` directory

**Today:** One file: `<html><body><h1>claude-proxy v2.0</h1><p>Dashboard coming soon.</p></body></html>`

**If standalone mode:** Replace with a real (minimal) login page. ~50 lines of HTML with CSS classes. Or keep it as a redirect to svg-terminal. Either way, trivial.

### `index.ts` startup

**Today:** `startApiServer()` is called with **6** options (no auth objects):

```typescript
startApiServer({ port, host, sessionManager, config, staticDir, dirScanner });
```

**If we wire auth + socket:** Becomes ~12 options (add `oauthManager`, `githubAdapter`, `userStore`, `provisioner`, `cookieSecret`, `cookieMaxAge`, `socketPath`). The `startApiServer` signature already accepts all the auth fields as optional — they just aren't passed.

## 4. What gets deprecated or removed

| Item | Current state | After this change |
|------|--------------|-------------------|
| **TCP `127.0.0.1:3101` as primary** | Default, only option | **Optional dev mode** — only listens if `api.port` is set in YAML |
| **`web/index.html` stub** | 1-line placeholder | Replaced with minimal login page or redirect |
| **`CLAUDE_PROXY_API` constant in `server.mjs`** | `'http://127.0.0.1:3101'` | Replaced with socket path constant (e.g. `CLAUDE_PROXY_SOCKET = '/run/claude-proxy/api.sock'`) |
| **Direct `fetch()` to claude-proxy in `server.mjs`** | 7 call sites | Replaced with `socketFetch()` wrapper (same HTTP semantics, different transport) |
| **Direct `new WebSocket('ws://127.0.0.1:...')` in `server.mjs`** | 2 call sites | Replaced with socket-path WebSocket connections |
| **OAuth wiring spec (`2026-04-03-oauth-web-api-wiring-spec.md`)** | Written for TCP-only model | **Partially superseded** — auth logic stays, but `public_base_url` concept changes (svg-terminal owns the public URL, not claude-proxy). Spec needs a revision note, not a full rewrite. |

**Nothing is deleted wholesale.** The auth modules, API routes, session management, and WebSocket streaming all stay. Transport changes; logic doesn't.

## 5. What stays exactly the same

| Component | Why it's unaffected |
|-----------|-------------------|
| **SSH transport (`:3100`)** | TCP — SSH clients connect from the network; unrelated to API transport |
| **All `/api/*` route handlers** | Same HTTP request/response semantics whether TCP or Unix socket |
| **WebSocket stream protocol** | Same JSON messages; transport changes, not protocol |
| **Session manager, PTY multiplexer, tmux** | No awareness of how the API server listens |
| **Auth modules (`src/auth/*`)** | Used by API routes; don't care about socket vs TCP |
| **Session form, lobby, widgets** | SSH-side; unrelated |
| **Launch profiles** (planned) | Can proceed independently |
| **Tests** | API tests use `http.request` which works with both TCP and Unix socket (test server can `listen(0)` for random TCP port as today) |

## 6. Security implications

| Concern | TCP loopback (today) | Unix socket (proposed) |
|---------|---------------------|----------------------|
| Any local user can connect to API | Yes — any process on the box | No — filesystem permissions restrict access |
| Accidental network exposure | Possible if someone changes `host: 0.0.0.0` | Not possible — Unix sockets are filesystem-only |
| Auth bypass | If `api.auth.required` isn't set, API is open | Same, but the surface is smaller (only processes with socket access) |
| Credential in transit | Cleartext on loopback (acceptable on same-box) | Cleartext in kernel memory (same security, no network stack) |

**Net:** Unix socket is **strictly more restrictive** than TCP loopback. No new attack surface.

## 7. The `ws` library and Unix sockets

Both repos use `ws@8.20.0`. The `ws` library supports custom `createConnection` on the client side:

```javascript
const ws = new WebSocket('ws+unix:///run/claude-proxy/api.sock:/api/session/id/stream');
```

Or via agent:

```javascript
import http from 'http';
const agent = new http.Agent({ socketPath: '/run/claude-proxy/api.sock' });
// ws doesn't directly use agents, but createConnection works
```

**Confirmed:** `ws@8` supports `ws+unix://` protocol URIs. No library change needed.

## 8. Open questions for your review

1. **Socket-only or dual-listen by default?** Option A: socket only unless `api.port` set. Option B: always both. Recommendation: A (socket only; TCP is dev opt-in).

2. **OAuth callback path:** Should svg-terminal's callback be at `/auth/callback` (svg-terminal namespace) or `/api/auth/callback` (matching claude-proxy's current route)? Recommendation: `/auth/callback` on svg-terminal — keeps svg-terminal routes clean and doesn't pretend to be claude-proxy.

3. **Cookie ownership:** svg-terminal sets its own cookie containing the signed token from claude-proxy. On each proxied API call, svg-terminal passes the token in an `Authorization` header (or custom header). claude-proxy validates the token signature. Does this model work for you, or do you prefer a shared cookie?

4. **Standalone login page:** Should claude-proxy's built-in login page be (a) a real usable page for people running claude-proxy without svg-terminal, or (b) just a redirect + error page? Recommendation: (a) — real but minimal, so claude-proxy is a complete product on its own.

5. **Timeline relative to launch profiles:** These are independent. Launch profiles (W3) can ship before, after, or in parallel with this. Any preference?

6. **Does the OAuth wiring spec need a full rewrite or just a revision note?** The spec's auth logic (modules, routes, ACL) is still correct. Only the transport and `public_base_url` concept changes. Recommendation: revision note at the top, not a rewrite.
