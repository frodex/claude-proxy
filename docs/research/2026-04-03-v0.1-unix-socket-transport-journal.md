## Context

claude-proxy and svg-terminal run on the same box. Today claude-proxy's HTTP/WS API listens on **TCP 127.0.0.1:3101**. svg-terminal connects to it via `http://127.0.0.1:3101`. The question: should claude-proxy expose a **Unix domain socket** instead of (or alongside) a TCP port?

---

## The architecture we're working toward

```
                    INTERNET / LAN
                         │
                    ┌────▼────┐
                    │  nginx  │   (optional reverse proxy — TLS, single public port)
                    └────┬────┘
                         │
               ┌─────────▼──────────┐
               │    svg-terminal    │   port 3200 (or behind nginx)
               │                    │
               │  • 3D dashboard    │
               │  • login screens   │
               │  • session mgmt    │
               │  • terminal cards  │
               └────────┬───────────┘
                        │
                Unix domain socket
              /run/claude-proxy/api.sock
                        │
               ┌────────▼───────────┐
               │   claude-proxy     │   NO public TCP port
               │                    │
               │  • SSH :3100       │   (SSH stays TCP — that's its transport)
               │  • session manager │
               │  • tmux sessions   │
               │  • auth / user DB  │
               └────────────────────┘
```

**Key point:** Browsers never talk to claude-proxy directly. svg-terminal is the only network-facing web service. claude-proxy's API is internal.

## Why Unix sockets (vs TCP loopback)

### What a Unix domain socket is

A socket **file** on the filesystem (e.g. `/run/claude-proxy/api.sock`). Processes connect to it like a network socket, but traffic never touches the network stack. It's how PostgreSQL, Docker, Redis, and nginx communicate with local backends.

### Comparison

| Concern | TCP 127.0.0.1:3101 | Unix socket |
|---------|---------------------|-------------|
| **Network exposure** | Loopback only — but any local process can connect | No network exposure at all |
| **Access control** | Firewall rules or app-level auth | **Filesystem permissions** (owner, group, mode) — same UGO model claude-proxy already uses for tmux sockets |
| **Port conflicts** | Possible — another service might want 3101 | Not possible — it's a file path |
| **Performance** | Good (loopback avoids NIC but still goes through TCP stack) | ~10-30% less overhead — no TCP headers, checksums, or congestion control |
| **Discoverability** | Port scanners find it; `ss -tlnp` shows it | `ls -la /run/claude-proxy/api.sock` — visible only to filesystem access |
| **Debugging** | `curl http://127.0.0.1:3101/...` just works | `curl --unix-socket /run/claude-proxy/api.sock http://localhost/...` — slightly more verbose |
| **Browser access** | Browsers can connect directly | Browsers **cannot** — requires a proxy or bridge |

### Why it fits this project

1. **You already use this pattern.** `SocketManager` creates tmux sockets at `/var/run/claude-proxy/sessions/*.sock` with UGO permissions. The API socket is the same concept for the HTTP/WS server.
2. **svg-terminal is the web UI.** Browsers connect to svg-terminal, not claude-proxy. So claude-proxy doesn't need a network port for web traffic.
3. **Security by default.** A misconfigured firewall or `host: 0.0.0.0` won't accidentally expose the API to the network.
4. **SSH stays TCP.** SSH clients need a real network port (`:3100`). That doesn't change — SSH is its own transport.

## What needs to change in code

### claude-proxy (server side)

**`src/api-server.ts`** — `server.listen()` call:

```
Today:   server.listen(port, host)        →  TCP 127.0.0.1:3101
Target:  server.listen(socketPath)        →  /run/claude-proxy/api.sock
```

Both modes can coexist (listen on socket AND optionally on TCP for dev). Node.js `http.createServer` supports both natively.

**Config (`claude-proxy.yaml`):**

```yaml
api:
  socket: /run/claude-proxy/api.sock     # primary — svg-terminal connects here
  socket_mode: 0660                       # owner + group can connect
  socket_group: cp-services              # svg-terminal's service user is in this group
  # port: 3101                           # optional — uncomment for dev/debug
  # host: 127.0.0.1
```

**Startup:** Ensure `/run/claude-proxy/` exists, unlink stale socket file, listen, `chown`/`chmod`.

### svg-terminal (client side)

**`server.mjs`** — replace TCP references:

```
Today:   fetch('http://127.0.0.1:3101/api/sessions')
Target:  http.request({ socketPath: '/run/claude-proxy/api.sock', path: '/api/sessions' })
```

For **WebSocket bridge** (`ws` library supports `socketPath`):

```javascript
const upstream = new WebSocket('ws+unix:///run/claude-proxy/api.sock:/api/session/' + id + '/stream');
```

(Exact syntax depends on the `ws` library version — may need `{ socket }` option instead.)

### Permissions

| File | Owner | Group | Mode | Effect |
|------|-------|-------|------|--------|
| `/run/claude-proxy/api.sock` | `root` | `cp-services` | `0660` | root and cp-services group members can connect |
| svg-terminal service user | — | member of `cp-services` | — | svg-terminal can reach the API |

Same pattern as tmux session sockets. No new concepts.

## The OAuth callback question

OAuth providers redirect the **browser** to a callback URL after login. Browsers can't connect to Unix sockets. Two options:

### Option A: svg-terminal handles the callback (recommended)

```
1. Browser clicks "Login with Google" on svg-terminal dashboard
2. svg-terminal redirects browser to Google with callback URL = https://dashboard.example.com/auth/callback
3. Google redirects browser back to svg-terminal with auth code
4. svg-terminal exchanges code with Google (server-side, standard OIDC)
5. svg-terminal calls claude-proxy (Unix socket) to resolve user identity + provision
6. svg-terminal sets its own session cookie
7. Browser is logged in
```

**Pros:** claude-proxy never needs a public port. All browser-facing auth lives in svg-terminal. One cookie domain.

**Cons:** OAuth code (openid-client, token exchange) needs to be in svg-terminal OR factored into a shared module. Currently it's in claude-proxy's `src/auth/oauth.ts`.

**Options for the auth code:**
- Move `OAuthManager` + `GitHubAdapter` into svg-terminal (they're ~150 lines total)
- OR svg-terminal forwards the raw code to claude-proxy via Unix socket, claude-proxy does the token exchange and returns the user identity. svg-terminal sets the cookie. Auth logic stays in claude-proxy. **This is the cleanest split.**

### Option B: Reverse proxy routes callback to claude-proxy

```
1. Browser clicks "Login with Google" on svg-terminal dashboard
2. Redirect to Google with callback URL = https://dashboard.example.com/api/auth/callback
3. Google redirects browser back
4. nginx routes /api/auth/callback → claude-proxy Unix socket
5. claude-proxy exchanges code, resolves user, sets cookie
6. claude-proxy redirects browser to svg-terminal dashboard
7. Browser is logged in (cookie set by claude-proxy domain)
```

**Pros:** Auth code stays in claude-proxy. No changes to existing auth modules.

**Cons:** Requires nginx (or similar) routing specific paths to different backends. Cookie domain must cover both services (same domain, different paths — fine if they're behind the same proxy). More moving parts.

### Recommendation

**Option A with forwarding.** svg-terminal receives the callback, passes the auth code to claude-proxy's internal API (`POST /internal/auth/exchange` over Unix socket), claude-proxy returns user identity, svg-terminal sets the cookie. Clean separation: **claude-proxy owns auth logic**, **svg-terminal owns the browser relationship**.

## What stays TCP

| Service | Port | Why |
|---------|------|-----|
| **SSH** | 3100 | SSH protocol requires TCP — clients connect from the network |
| **svg-terminal** | 3200 (or 443 behind proxy) | Browsers need a network endpoint |
| **claude-proxy API** (dev only) | 3101 | Optional — `curl` convenience, disabled in production |

## Implementation effort

| Component | Size | Notes |
|-----------|------|-------|
| `api-server.ts` listen on socket | Small | ~10 lines: unlink stale, `listen(path)`, `chmod`/`chown` |
| Config extension | Small | `api.socket`, `api.socket_mode`, `api.socket_group` |
| svg-terminal `fetch` via socket | Medium | Replace `fetch()` calls with `http.request({ socketPath })` or a thin wrapper |
| svg-terminal WS bridge via socket | Medium | `ws` library socket path option |
| OAuth callback forwarding | Medium | New internal endpoint on claude-proxy; svg-terminal callback handler |
| Socket permissions on startup | Small | Same pattern as `SocketManager` |

## Open questions

1. **Should TCP dev mode be a flag or always available?** Simplest: if `api.port` is set in YAML, listen on both socket and TCP. If not, socket only.
2. **Does svg-terminal need its own session cookie, or can it share claude-proxy's?** If same domain behind one proxy, one cookie works. If separate origins, each needs its own.
3. **Does the `ws` library version in svg-terminal support Unix socket paths natively?** Needs a quick check of the installed version.
