## Question 1: Sequential wizard vs random-access menu for edit mode

### The problem

The original plan said "replace startEditSession with SessionForm" — swapping a menu-based edit flow (pick which field to change) with a sequential FlowEngine walk-through (advance through all fields in order). FlowEngine had no backward navigation or field jumping.

### What I asked

Is edit mode supposed to be sequential (walk through all fields even if you only want to change one) or random-access (jump to any field)?

### Authoring agent's answer

Sequential with backward navigation. User sees all fields (full-form rendering) but advances one by one. Added: goBack(), validation state, submit gate with RED flagging. User added: "arrow keys like lobby" — free cursor movement between fields.

### My concern

Arrow up/down conflict. ListPicker and CheckboxPicker use arrows for internal navigation. If arrows also move between fields, we have a collision. Three approaches considered:

- **A:** Enter confirms + advances, arrows stay within widget. Simple but can't freely navigate fields.
- **B:** Arrows move between fields, Enter opens widget for editing. Two-mode but matches lobby mental model.
- **C:** Tab/Shift+Tab between fields, arrows stay with widget. Standard form convention but Tab already means tab-completion.

### User's decision

**Two-mode interaction with color signaling:**
- **Field navigation mode (yellow):** Arrows move between fields. Enter activates the selected field's widget.
- **Widget edit mode (white/bold):** All keys including arrows go to the widget normally. Enter confirms selection and returns to field nav. Escape cancels and returns to field nav.
- Yellow → white color shift on the active field label signals which mode you're in.

### Why this works

Eliminates the arrow conflict entirely. Each widget gets its full key handling in edit mode. The color shift is a clear visual indicator — no guessing which mode you're in. Matches the lobby pattern (arrows move cursor) with an explicit activation step (Enter to edit).

### What this adds to the plan

FlowEngine needs a `mode: 'navigate' | 'edit'` state. `handleKey()` routes based on mode. In navigate mode, arrows move a field cursor. Enter toggles to edit mode. In edit mode, all keys delegate to the active widget. Enter/Escape toggle back to navigate mode. Renderers check mode + field state to pick colors (yellow for navigate-selected, white for editing, green for completed, etc).

## Question 2: Display values — _display_ keys vs displayValue callback

### The problem

`getFlowSummary()` needs to show completed field values (e.g., "my-project" next to "Session name"). But step IDs don't match accumulated keys — 8 out of 11 steps have different IDs vs accumulated keys (e.g., step `runas` → `acc.runAsUser`). The plan proposed `_display_<stepId>` parallel keys in accumulated.

### What I proposed

A `displayValue` callback on FlowStep:

```typescript
displayValue?: (accumulated: Record<string, any>) => string;
```

Each step defines how to display its result. No parallel data path, no naming convention.

### Decision

**Accepted — use displayValue callback.** Authoring agent agreed. User didn't object. Fallback: `String(accumulated[step.id])` if no callback provided.

### Why

The `_display_` convention is fragile — depends on every `onResult` remembering to write a second key with a specific naming pattern. A callback is explicit, co-located with the step definition, and can format complex values (arrays → comma-separated, booleans → Yes/No, etc).

## Question 4 (emerged during review): YAML-driven form config

### The problem

The authoring agent's reply directed: "field configuration should be YAML-driven if possible." This wasn't in the original plan.

### My assessment

Wrote an impact report (`2026-03-31-yaml-form-config-impact.md`). Findings:
- YAML parser already a dependency (`yaml@2.8.3`)
- Config loader with deepMerge already exists (`src/config.ts`)
- ~305 lines of new code
- Widget creation + onResult stay in code (need runtime data). YAML owns metadata: field order, labels, locked/visible/prefill per mode.
- Unlocks: add modes without code, reorder fields via config, per-deployment customization, future web form generation from same config.

### User's decision

**Approved.** "It's likely worth it" — designing for reuse opens flexibility for changes without recoding.

### Risk noted

Accumulated-state conditions (`not-hidden`, `not-hidden-and-not-public`) reference other field values at runtime. YAML names the condition, code defines the predicate function. This is the one non-trivial YAML↔code boundary.

## Question 5 (emerged during review): Edit-mode runas semantics

### The problem

The corrected matrix had `runas` as active (editable) on edit and restart. But changing which UID a running tmux process runs under requires `setuid()` — effectively a process restart. The edit form implies lightweight metadata changes, not process lifecycle operations.

### What I found

- Changing socket ownership (`chown`/`chmod` on tmux socket): **possible** at runtime. Changes who can connect.
- Changing process UID: **not possible** without kill + relaunch.
- The UGO/socket permission system could allow access control changes, but that's different from changing who the process runs *as*.

### User's decision

**Lock runas on edit/restart/fork.** Only editable on create (where the process is freshly launched). Unlocking in the future would require a session export + migration feature to move the session to the other user's home directory. This is explicitly deferred — not in scope for this plan.
